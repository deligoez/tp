# tp audit — Post-Implementation Spec Review

## Problem

tp has `tp lint`, `tp review`, `tp validate`, and `tp report` — but all operate pre-implementation or mid-implementation. There is no mechanism to answer "did what we wrote in the spec actually make it into the code?" after implementation completes. This is done manually by spawning a sub-agent.

## Solution

New command: `tp audit <spec.md>`

Parses the spec's structured elements (table rows, numbered list items, code snippets), reads changed source files, and generates adversarial prompts that verify each requirement against actual code.

## Design Principles

### AX-1: Zero-config for agents

Agent should not have to manually specify which files changed. `tp audit` auto-detects changed files via `git diff`, eliminating an extra round-trip.

### AX-2: Checklist output, not findings

`tp review` produces NDJSON findings (pre-implementation spec critique). `tp audit` produces a checklist with per-requirement status: `pass`, `partial`, `fail`. This is a different mental model — the agent reads the checklist and decides "all pass, ready to merge" vs "3 failures, need fixes."

### AX-3: Structured element granularity

Every table data row, numbered list item, and task acceptance criterion becomes a checklist entry. No vague "looks good" — each spec requirement gets explicit verification.

### AX-4: Reuse review infrastructure

File reading, caps, budget enforcement — all reuse `readAffectedFiles` from `review.go`. Prompt generation follows same patterns. No duplicated infrastructure.

## Command Interface

```bash
tp audit <spec.md>                          # Auto-detect changed files via git diff
tp audit <spec.md> --affected-files a.go b.go  # Manual override
tp audit <spec.md> --base main              # Diff against specific ref
tp audit <spec.md> --base HEAD~3            # Last 3 commits
tp audit <spec.md> --findings review.ndjson # Also verify review findings were fixed
tp audit <spec.md> --json                   # JSON output
tp audit <spec.md> --compact                # Minimal JSON
```

### Flags

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--affected-files` | StringArray | auto-detect | Source files to audit (auto-detect via git diff if omitted) |
| `--base` | string | — (staged+unstaged) | Git ref to diff against. When set: `git diff <base>...HEAD`. When omitted: staged + unstaged changes |
| `--findings` | string | — | Path to NDJSON findings from `tp review` — verify they were addressed. Reads `finding` field for text; falls back to `message` or `description` |
| `--json` | bool | auto | Force JSON output |
| `--compact` | bool | false | Minimal JSON: truncates checklist text to 80 chars, omits file_summary |

### Behavior

1. Parse spec: extract structured elements (tables, numbered lists, code blocks) + task acceptance from `.tasks.json` if present
2. Determine files to audit:
   - If `--affected-files` provided: use those (with dedup and validation, same as review)
   - If omitted: run `git diff --name-only <base>` (default: staged+unstaged changes) — filter to non-binary, non-markdown files, cap at 50 files
3. Build checklist entries: one entry per table row, numbered list item, task acceptance criterion
4. If `--findings` provided: add entries for each finding ("finding #N addressed?")
5. Read affected files (reuse `readAffectedFiles` with same caps: 8000/file, 50000 total). Budget calculation: spec_content (up to specContentCap=10000) + checklist_text (estimated) + findings_summary (up to 5000 if --findings) + source_files. Total must not exceed promptBudget=60000. Reduce source files from end if budget exceeded.
6. Generate 1-2 adversarial prompts:
   - Prompt 1: "Verify each spec requirement against the code. For each checklist item, mark pass/partial/fail with evidence."
   - Prompt 2 (if findings provided): "Each review finding was reported. Verify it was addressed in the code. Mark pass/fail."
7. Output: JSON with `checklist` array + `prompts` array

### Auto-Detection via Git Diff

```bash
# Default: staged + unstaged changes (no --base)
tp audit spec.md
# runs: git diff --name-only (unstaged) + git diff --name-only --cached (staged)

# Against a specific ref
tp audit spec.md --base main
# runs: git diff --name-only main...HEAD

# Last N commits
tp audit spec.md --base HEAD~5
# runs: git diff --name-only HEAD~5...HEAD
```

Filtering rules for auto-detected files:
- Skip binary files (no extension or common binary extensions: .png, .jpg, .gif, .svg, .ico, .woff, .woff2, .ttf, .eot, .zip, .tar, .gz, .pdf, .exe, .dll, .so, .dylib, .o, .a)
- Skip markdown files (.md) — spec itself should not be in the audit
- Skip .tasks.json files — task metadata, not code
- Cap at 50 files — if more, take the first 50 and add an info-level message
- Sort by path for determinism

### Git Availability

If not in a git repo or `git` is unavailable:
- If `--affected-files` provided: proceed as normal
- If auto-detect and no git: error with hint "not in a git repo — provide --affected-files or run inside a git repo"
- Test guidance: tests use `t.TempDir()` (not a git repo) to test the git-not-available path. Tests use `--affected-files` for all other test cases.

## Checklist Entry Format

Each structured element becomes a checklist entry:

```json
{
  "id": "table-3-2",
  "type": "table_row",
  "spec_line": 45,
  "section": "API Endpoints",
  "text": "POST /users — create user endpoint with validation",
  "status": null,
  "prompt": 0
}
```

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique ID: `table-<table_idx>-<row_idx>`, `list-<list_idx>-<item_num>`, `task-<task_id>`, `finding-<n>` |
| `type` | string | One of: `table_row`, `list_item`, `task_acceptance`, `finding` |
| `spec_line` | int | Line number in spec (0 for findings) |
| `section` | string | Parent heading text |
| `text` | string | The requirement text (row content, list item text, acceptance string, finding message) |
| `status` | string | `null` (unchecked) — filled by agent after running prompt |
| `prompt` | int | Which prompt (0-indexed) this entry appears in |

### Entry Generation Rules

**Table rows:**
- For each table in `structured_elements.tables`, generate one entry per data row
- `text` = full row content (pipe-separated cells, trimmed)
- `id` = `table-{i}-{j}` where i = table index, j = row index within table

**Numbered list items:**
- For each numbered list in `structured_elements.numbered_lists`, generate one entry per item
- `text` = full item text (number + content)
- `id` = `list-{i}-{num}` where i = list index, num = item number (#1, #2, etc.)

**Code blocks:** Not included as checklist entries. Code blocks in specs are illustrative (examples, API contracts), not requirements. Tables and numbered lists capture the actual requirements. This is intentional — including code blocks would create noise without verification value.

**Task acceptance:**
- If `.tasks.json` exists alongside spec, generate one entry per task
- `text` = task acceptance string
- `id` = `task-{task_id}`
- `section` = task title

**Review findings:**
- If `--findings` provided, read NDJSON line by line, generate one entry per finding
- `text` = value of first non-empty field among: `finding`, `message`, `description`, `title` (in that priority order). If all empty, skip the finding with a warning.
- `id` = `finding-{n}` where n = sequential finding index (only counting non-skipped findings)
- `section` = "Review Findings"

```json
      "status": null,

    }

  "prompts": [
    {
      "role": "implementation-auditor",
      "prompt": "...",

    }
  ]

```

### Compact JSON

`--compact` omits: `file_summary`. Checklist `text` truncated to 80 chars. Prompts still included (they are the primary output).

## Prompt Template

### Prompt 0: Implementation Verification

```
Role: implementation-auditor
Task: Verify each spec requirement is implemented in the code.

## Spec Excerpt
<spec content, capped at specContentCap>

## Checklist
For each item, mark it as:
- PASS: requirement is fully implemented in the code
- PARTIAL: requirement is partially implemented — explain what's missing
- FAIL: requirement is not found in the code — explain what's expected

<checklist items with IDs and text>

## Source Files
<affected files content, capped per-file and total>

## Rules
1. Read the code carefully. State-dependent behaviors (disabled states, loading conditions, conditional rendering) count as partial unless fully covered.
2. A table row describing a feature is PASS only if the feature code exists AND handles edge cases mentioned in surrounding spec context.
3. A numbered list item describing a test is PASS only if a corresponding test function exists with assertions covering the described behavior.
4. Task acceptance criteria are PASS only if the described behavior is observable in the code (not just a comment or placeholder).
5. If a requirement mentions specific error handling, validation, or edge cases, verify those exist — don't just check the happy path.

## Output Format
For each checklist item, output one line:
{ID} | {PASS|PARTIAL|FAIL} | {evidence — file:line reference or explanation}
```

### Prompt 1: Review Findings Verification (if --findings provided)

```
Role: implementation-auditor
Task: Verify each review finding was addressed in the code.

## Review Findings
<findings from NDJSON, summarized>

## Checklist
<finding checklist items>

## Source Files
<affected files content>

## Rules
1. For each finding, determine if the code change addresses the reported issue.
2. A finding is PASS only if the specific problem described is demonstrably fixed.
3. Partial fixes (e.g., adding a comment instead of actual code) count as PARTIAL.
4. If multiple findings relate to the same code area, verify each independently.

## Output Format
{ID} | {PASS|PARTIAL|FAIL} | {evidence — file:line reference or explanation}
```

## Prompt Distribution

When there are many checklist entries (>=50), split across prompts:
- Prompt 0: first 50 entries (or all entries if < 50)
- Prompt 1: next 50 entries (or findings prompt if --findings provided and no overflow)
- Each prompt gets the full source files (not split — code context must be complete)

### Empty Checklist

If the spec has no structured elements and no task file:
- Output a warning: "no structured elements found in spec — checklist is empty"
- Return empty checklist array with summary `total: 0`
- Exit code 0 (not an error — spec may be prose-only)

## Error Handling

| Condition | Exit Code | Message |
|-----------|-----------|---------|
| No spec argument | 2 | "spec path required" |
| Spec file not found | 3 | "spec not found: {path}" |
| Not in git repo, no --affected-files | 4 | "not in a git repo — provide --affected-files or run inside a git repo" |
| No changed files detected, no --affected-files | 4 | "no changed files detected (diff {base}...HEAD is empty) — provide --affected-files" |
| All files filtered out (only binary/markdown) | 4 | "no audit-able files in diff — only binary/markdown files changed" |

## Exit Codes

Reuses existing exit code conventions: 0=success, 2=usage, 3=file, 4=state.

## Interaction with Existing Features

| Feature | Interaction |
|---------|-------------|
| `--affected-files` | Same validation, dedup, caps as `tp review --affected-files` |
| `--findings` | Reads NDJSON in same format as `tp review --round N --findings` |
| `--json` / `--compact` | Standard output mode (same as all commands) |
| `--file` | Standard task file discovery (for reading task acceptance) |
| `tp lint` | Audit uses `CheckStructuredElements` internally — no duplication |
| `tp review` | Audit is post-implementation; review is pre-implementation — complementary |

### Infrastructure Reuse Strategy

`readAffectedFiles`, `buildAffectedSummary`, `buildAffectedSection`, and `dedupPaths` are currently private in `review.go`. Rather than duplicating them in audit.go:

1. Extract to `internal/engine/fileio.go` as exported functions
2. Update `review.go` to call `engine.ReadAffectedFiles(...)` etc.
3. `audit.go` calls the same engine functions
4. No behavioral change — pure refactoring, test-verified by existing review tests
