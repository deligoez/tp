# Review Orchestration — Efficient Multi-Round Workflow

**Status:** Draft
**Date:** 2026-04-04

## 1. Problem

`tp review` generates high-quality adversarial prompts but leaves all orchestration to the agent. Two independent agents reported identical friction after 6-8 round review sessions:

1. **Manual loop per round:** `tp review` → parse prompts → spawn 3 agents → wait → merge findings → fix spec → repeat. This 6-step loop is performed N times (typically 6-8).
2. **Full spec repeated in every prompt.** 800-line spec × 3 personas × 6 rounds = 18 prompts carrying ~90K tokens of spec content. After R1, ~80% of the spec is unchanged.
3. **Findings summary grows unbounded.** R6 prompt carries 75+ previous findings (~5K tokens per prompt). Most are already fixed — only the delta matters.
4. **Finding merge is manual.** `cat f0.ndjson f1.ndjson f2.ndjson > all.ndjson` between rounds.
5. **Convergence detection is manual.** Agents tracked "R1=56, R2=23, R4=3, R5=0, R6=0" by hand.
6. **No finding resolution tracking.** Which findings are fixed, which are won't-fix — no mechanism.
7. **No cross-round report.** Convergence chart and summary built manually.
8. **Late rounds are wasteful.** R5-R6 are "is anything still broken?" but run full-weight review prompts.

Estimated token savings from addressing these: **40-50%** of total review workflow tokens.

## 2. Root Cause

`tp review` is a **prompt generator**, not a **workflow tool**. It optimizes the single-call case (P4: Agent Plans, Tool Executes) but creates compounding overhead when the agent must orchestrate N calls. The tool does not accumulate state across rounds — each round starts fresh.

## 3. Proposed Solution

Seven features that transform multi-round review from a manual loop into an efficient workflow:

| Feature | Flag/Command | Purpose |
|---------|-------------|---------|
| A | `tp review --merge` | Merge + dedup findings from multiple NDJSON files |
| B | `tp review --resolve` | Mark findings as fixed/wontfix with evidence |
| C | Compact findings summary | Only carry unresolved findings + "N resolved" count |
| D | Diff-based review | `--diff-from <path>` sends only changed spec lines in R2+ |
| E | Lightweight verification | `--verify` generates minimal "still broken?" prompt |
| F | `tp review --report` | Cross-round convergence report |
| G | Spec reference mode | `--spec-ref` omits inline spec, tells agent to read file |

## 4. Design Principles

| # | Principle | Application |
|---|-----------|-------------|
| P1 | AX First | Every feature reduces agent round-trips or manual steps |
| P2 | Batch Parity | Merge, resolve, report work on N findings as easily as 1 |
| P3 | Minimal Tokens | Diff-based, compact summary, spec-ref, verify mode all cut tokens |
| P4 | Agent Plans, Tool Executes | tp stays deterministic — no agent spawning, no state machine |

### What Doesn't Change

- tp does NOT become an orchestrator. It does NOT spawn agents or manage state machines.
- tp does NOT auto-fix specs based on findings.
- The 3-persona adversarial review structure stays the same.
- `tp review` without new flags is 100% backward compatible.
- All output is JSON when piped.

### Mode Mutual Exclusivity

Features A (merge), B (resolve), E (verify), and F (report) are separate **modes** of the review command. They are mutually exclusive with each other and with the default 3-persona review mode. Only one mode can be active per invocation:

- `--merge` — merge mode
- `--resolve` / `--resolve-all` — resolve mode
- `--verify` — verification mode
- `--report` — report mode
- (none of the above) — default review mode

Flags `--diff-from`, `--spec-ref`, `--round`, `--findings`, `--affected-files`, `--final-round` apply to the default review mode and verify mode (where applicable). They are rejected with merge, resolve, and report modes. Specifically:
- `--final-round` (existing flag): compatible with default review mode only. Not compatible with `--verify` (verify serves a different purpose than final-round).
- `--diff-from`, `--spec-ref`, `--affected-files`: compatible with default review mode and `--verify`.
- `--round`, `--findings`: compatible with default review mode only (verify does not use round numbers).

## 5. Feature A: Finding Merge (`--merge`)

### 5.1 Command

```bash
tp review --merge f1.ndjson [f2.ndjson ...] [-o merged.ndjson]
```

**Implementation note:** `--merge`, `--resolve`, and `--report` are boolean flags that switch the command into a different mode. In each mode, positional arguments are reinterpreted:
- `--merge`: positional args = NDJSON file paths (no spec file)
- `--resolve`: positional args = [file, index, status, evidence] (no spec file)
- `--report`: positional args = NDJSON file paths or directory (no spec file)
- `--verify`: still requires spec file as first positional arg

Cobra's `Args` validator and `RunE` function detect which mode is active and parse positional args accordingly. `--force` is a standard Cobra boolean flag parsed independently of positional args.

### 5.2 Behavior

1. Read all input NDJSON files sequentially.
2. Parse each line as a finding object. A valid finding must have at least `severity` and `finding` fields. Empty/blank lines are silently skipped. Lines that fail JSON parse or lack required fields are skipped with a warning to stderr.
3. Extra fields (including `resolved`) are preserved through merge.
4. Deduplicate by `(category, location, finding_prefix)` where `finding_prefix` is the first 80 characters of the `finding` field. This prevents discarding different findings that share the same category and location. When duplicates are found, keep the one with the highest severity (critical > high > medium > low > unknown). Ties keep the first occurrence.
5. Sort by severity (critical > high > medium > low > unknown), then category alphabetically.
6. Write merged NDJSON to stdout (default) or to `-o` path if provided.
7. Print summary to stderr: `merged: N unique findings from M files (K duplicates removed)`.

### 5.3 Validation

| Condition | Exit Code | Message |
|-----------|-----------|---------|
| No input files | 2 | "at least 1 file required for merge" |
| Input file not found | 3 | "file not found: {path}" |
| All lines invalid | 1 | "no valid findings in input files" |

Note: A single file is valid — merge with 1 file normalizes (dedup + sort) the findings.

### 5.4 Output Format

Default output: merged NDJSON to stdout. Each line:
```json
{"severity":"high","category":"completeness","location":"section 5","finding":"...","suggestion":"..."}
```

When `-o` is provided: merged NDJSON goes to the `-o` file. Stdout receives the JSON summary:
```json
{
  "merged_count": 42,
  "input_files": 3,
  "duplicates_removed": 8,
  "output_path": "merged.ndjson",
  "by_severity": {"critical": 2, "high": 15, "medium": 20, "low": 5}
}
```

When `--json` without `-o`: merged NDJSON is included as a `findings` array in the JSON result:
```json
{
  "merged_count": 42,
  "input_files": 3,
  "duplicates_removed": 8,
  "output_path": "stdout",
  "by_severity": {"critical": 2, "high": 15, "medium": 20, "low": 5},
  "findings": [{"severity":"high", ...}, ...]
}
```

When `--json` WITH `-o`: `-o` file always receives raw NDJSON. Stdout receives the JSON summary (same as `-o` without `--json` — the `--json` flag is redundant when `-o` is present).

TTY behavior: merge always outputs raw NDJSON to stdout regardless of TTY status. This is an intentional exception to tp's usual TTY convention (colored text in TTY, JSON when piped) because merge is a data operation — the NDJSON output IS the result, and adding a separate TTY presentation would require a lossy rendering of findings. The stderr summary provides human-readable feedback.

## 6. Feature B: Finding Resolution (`--resolve`)

### 6.1 Command

```bash
tp review --resolve <findings.ndjson> <finding-index> <status> [evidence]
tp review --resolve findings.ndjson 3 fixed "addressed in section 11.2"
tp review --resolve findings.ndjson 7 wontfix "intentional — out of scope per design"
tp review --resolve findings.ndjson 7 duplicate "same as finding 3"
tp review --resolve-all findings.ndjson fixed "all addressed in spec revision"
tp review --resolve findings.ndjson 3 fixed --force   # re-resolve already resolved
```

### 6.2 Behavior

1. Read findings NDJSON file.
2. Parse lines into findings array (0-indexed).
3. Add/update `resolved` field on the target finding:
   ```json
   {"severity":"high","finding":"...","resolved":{"status":"fixed","evidence":"addressed in section 11.2","resolved_at":"2026-04-04T14:30:00Z"}}
   ```
   For `duplicate` status, the evidence field should reference the original finding (e.g., "same as finding 3" or "duplicate of category:location key").
4. Acquire flock on the findings file before read-modify-write (consistent with all other tp write operations).
5. Write updated NDJSON back to the same file (in-place update).
6. `--resolve-all` applies the same status to ALL unresolved findings. Already-resolved findings are not modified unless `--force` is also provided.
7. When evidence is omitted, it defaults to empty string `""` in the resolved object.

### 6.x Output

On success, print confirmation to stderr:
```
resolved finding {N} as {status}
```

When `--json` is active, output to stdout:
```json
{"index": 3, "status": "fixed", "evidence": "addressed in section 11.2", "file": "findings.ndjson"}
```

For `--resolve-all`, stderr shows `resolved {N} findings as {status} ({M} already resolved, skipped)`. JSON output includes `{"resolved_count": N, "skipped_count": M, "status": "fixed", "file": "findings.ndjson"}`.

### 6.3 Resolution Statuses

| Status | Meaning |
|--------|---------|
| `fixed` | Finding was addressed in spec or code |
| `wontfix` | Intentionally not addressed (with reason) |
| `duplicate` | Same as another finding (evidence should reference original) |

### 6.4 Validation

| Condition | Exit Code | Message |
|-----------|-----------|---------|
| File not found | 3 | "findings file not found: {path}" |
| Invalid index | 2 | "finding index {N} out of range (0-{max})" |
| Invalid status | 2 | "invalid status: {s} (must be fixed, wontfix, or duplicate)" |
| Already resolved (no --force) | 1 | "finding {N} already resolved as {status}" with hint "use --force to re-resolve" |

`--force` flag: when present, allows re-resolving an already-resolved finding, updating its status, evidence, and resolved_at timestamp.

### 6.5 Impact on Findings Summary

When `tp review --round N --findings` reads a findings file:
- Findings with `resolved.status = "fixed"` are excluded from the detailed "DO NOT re-report" listing but counted in the compact resolved summary line (see Feature C, section 7.2).
- Findings with `resolved.status = "wontfix"` are included in the detailed listing with a `[WONTFIX]` tag.
- Findings with `resolved.status = "duplicate"` are excluded entirely (they are redundant by definition).
- Unresolved findings (no `resolved` field) are included in the detailed listing as before.

## 7. Feature C: Compact Findings Summary

### 7.1 Current Behavior

`buildFindingsSummary` includes ALL previous findings in the prompt, sorted by severity. At R6 with 75+ findings, this is ~5K tokens per prompt.

### 7.2 New Behavior

Summary is split into two sections:

**Section 1: Unresolved + wontfix findings (full detail)**
```
UNRESOLVED findings from previous rounds — DO NOT re-report:
  [HIGH] completeness — section 5: missing error handler for batch timeout
  [WONTFIX] ambiguity — section 7.2: "appropriate" retry count is undefined (wontfix: out of scope)
```

**Section 2: Resolved findings (count only)**
```
Additionally, {N} findings from previous rounds were RESOLVED (fixed or duplicate) and are no longer active.
Do not re-report resolved issues. Focus ONLY on NEW issues in the current spec.
```

### 7.3 Token Impact

- R1: No change (no previous findings).
- R2: Minor change (few findings).
- R4+: Dramatic reduction. 75 findings → maybe 5 unresolved + "70 resolved" one-liner.

### 7.4 Backward Compatibility

If findings file has no `resolved` field on any entry, all findings are treated as unresolved — same behavior as today.

## 8. Feature D: Diff-Based Review (`--diff-from`)

### 8.1 Command

```bash
tp review spec.md --round 2 --findings r1.ndjson --diff-from spec-r1.md
```

### 8.2 Behavior

1. Read current spec (`spec.md`) and baseline spec (`spec-r1.md`).
2. Compute section-level diff (added/modified/deleted sections).
3. Generate review prompts with TWO sections:
   - **Changed sections** (full text) — "Review these changes carefully"
   - **Unchanged sections** (headings only) — "These sections were not modified. Only flag issues if they interact with the changes above."
4. Structured elements from changed sections get full "check each row" treatment.
5. Structured elements from unchanged sections are listed by heading only.

### 8.3 Diff Format in Prompt

```
## Changed Sections (review carefully)

### 5.2 Batch Processing (MODIFIED)
[full section text]

### 7.1 New Error Codes (ADDED)
[full section text]

## Removed Sections

- "6.3 Legacy Support" was removed from the spec.

## Unchanged Sections (review only if interacting with changes)

- 1. Introduction
- 2. Design Principles (4 subsections)
- 3. Command Interface (3 tables, 2 lists)
- 4. Validation Rules (1 table)
- 6. Output Format (2 code blocks)
```

### 8.4 Token Savings

For an 800-line spec where R2 changed 20%:
- Current: 800 lines × 3 prompts = 2400 lines of spec text
- Diff-based: 160 lines full + 640 lines as headings (~80 lines) × 3 = 720 lines
- **~70% reduction per round for R2+**

Note: Savings depend on section granularity. If a single line changes in a 200-line section, the entire section is included. For large monolithic sections, savings are reduced.

### 8.5 Validation

| Condition | Exit Code | Message |
|-----------|-----------|---------|
| `--diff-from` file not found | 3 | "diff baseline not found: {path}" |
| `--diff-from` without `--round >= 2` | 2 | "--diff-from requires --round >= 2" |
| No differences detected | 0 | Output includes hint: "no changes detected between baseline and current spec — review may be unnecessary" |

### 8.6 Diff Algorithm

Use section-level comparison:

1. Parse both specs into section trees using `ParseHeadings`. A "section" is a heading line plus all content lines until the next heading of equal or higher level.
2. Content inside fenced code blocks (``` ... ```) is included in the section content but heading-like lines within code blocks are NOT treated as section boundaries.
3. Match sections between baseline and current spec by heading text (exact match after trimming).
4. For matched sections, compare content line-by-line after trimming leading/trailing whitespace per line. Blank line count differences within a section are ignored. A section is "changed" if any non-blank content line differs.
5. A section in the current spec with no matching heading in the baseline is "added".
6. A section in the baseline with no matching heading in the current spec is "deleted" — mentioned as "Removed: {heading}".

**Limitation:** Renamed headings (e.g., "5.2 Batch Processing" → "5.2 Batch Operations") are treated as one deletion plus one addition. No rename detection is performed.

## 9. Feature E: Lightweight Verification (`--verify`)

### 9.1 Command

```bash
tp review spec.md --verify --findings all-rounds.ndjson [--affected-files ...] [--diff-from ...]
```

### 9.2 Purpose

After multiple review rounds with convergence (0 new findings), a final lightweight check: "are all previous findings genuinely resolved, and are there any new critical issues?"

### 9.3 Behavior

1. Generate a SINGLE prompt (not 3 personas) with role `verifier`.
2. Read findings file and classify: resolved (fixed/duplicate), wontfix, and unresolved.
3. Prompt structure adapts to findings state:
   ```
   You are a verification reviewer. Previous review rounds produced {N} findings:
   - {M} were addressed (fixed or duplicate)
   - {W} were marked won't-fix
   - {U} remain unresolved

   Your job is NOT to do a full review — only:

   1. Verify each fixed/duplicate finding is genuinely resolved in the spec.
   2. For unresolved findings, confirm they are still present.
   3. Check if any fix introduced a NEW critical or high-severity issue.
   4. If everything checks out, respond with an empty array.

   Fixed findings to verify:
   [list of fixed findings with their resolution evidence]

   Won't-fix findings (acknowledged, not bugs):
   [list of wontfix findings]

   Unresolved findings (still open):
   [list of unresolved findings]

   Spec content:
   [full spec, diff-based spec, or spec-ref depending on flags]

   [Affected files section if --affected-files provided]

   Respond with one JSON finding per line (NDJSON). If no issues found, respond with zero lines (empty output).
   For findings that were reported as fixed but are NOT actually resolved, set category to "regression".
   ```
   Categories with 0 entries are omitted from the prompt entirely (e.g., if there are no wontfix findings, the "Won't-fix findings" section is not generated). This saves tokens.
4. When `--affected-files` is provided, affected file content is included in the verify prompt (same budget rules as default review).
5. When `--diff-from` is provided, verify prompt uses diff-based spec content.
6. The `review_loop` output includes `"mode": "verification"` to distinguish from full review. For non-verify modes (default review, perspectives), `mode` is omitted.

### 9.4 Token Savings

- Full review: 3 prompts × full spec + findings summary = ~45K tokens
- Verification: 1 prompt × compact findings + spec = ~12K tokens
- **~73% reduction per verification round**

### 9.5 Validation

| Condition | Exit Code | Message |
|-----------|-----------|---------|
| `--verify` without `--findings` | 2 | "--verify requires --findings (nothing to verify without previous findings)" |
| `--verify` with `--perspective` | 2 | "--verify is mutually exclusive with --perspective" |
| `--verify` with `--round` | 2 | "--verify is mutually exclusive with --round (verification is not a numbered round)" |

### 9.6 Output

Same `reviewResult` structure but:
- `prompts` array has 1 entry (not 3) with role `verifier` and category `verification`.
- `review_loop.mode` is `"verification"`.
- `review_loop.instruction` says "If verifier finds 0 issues, review is complete. If issues found, run a full review round."

Expected output schema for verification findings is the same as review findings: `{severity, category, location, finding, suggestion}`.

## 10. Feature F: Review Report (`--report`)

### 10.1 Command

```bash
tp review --report r1.ndjson r2.ndjson r3.ndjson [r4.ndjson ...]
tp review --report rounds/           # reads all .ndjson files in directory, sorted by name
```

Note: `--report` does NOT take a spec file argument. Input files must be per-round findings (one file = one round). Do not pass merged files.

### 10.2 Output (TTY)

```
Review Convergence Report
═════════════════════════

Round  In File  New  Resolved  Unresolved  Δ
  R1       46    46         0          46   —
  R2       23    12        35          23  -50%
  R3       20     8        11          20  -13%
  R4        3     1        18           3  -85%
  R5        0     0         3           0 -100%
  R6        0     0         0           0    0%

Status: CONVERGED (2 consecutive clean rounds)

By severity:
  critical:  4 found,  4 fixed,  0 remaining
  high:     18 found, 18 fixed,  0 remaining
  medium:   31 found, 28 fixed,  3 wontfix
  low:      21 found, 21 fixed,  0 remaining

By category:
  completeness: 28 | ambiguity: 19 | consistency: 14 | feasibility: 8 | redundancy: 5
```

Column definitions:
- **In File:** Total findings in that round's NDJSON file.
- **New:** Findings whose `(category, location, finding_prefix)` key does not appear in the UNION of all previous rounds' key sets.
- **Resolved:** Findings whose key was present in the immediately preceding round but is absent in the current round. Each finding is counted as resolved only once — when it first disappears. A finding cannot be double-counted across rounds.
- **Unresolved:** Cumulative unresolved findings after this round = `previous_unresolved + new - resolved`. For R1, `previous_unresolved` is 0, so `unresolved = new`.
- **Δ:** Percentage change in unresolved count vs previous round. `(current - previous) / previous * 100`. Dash for R1.

### 10.3 Output (JSON)

```json
{
  "rounds": [
    {"round": 1, "in_file": 46, "new": 46, "resolved": 0, "unresolved": 46, "delta_percent": null},
    {"round": 2, "in_file": 23, "new": 12, "resolved": 35, "unresolved": 23, "delta_percent": -50.0}
  ],
  "convergence": {
    "converged": true,
    "consecutive_clean": 2,
    "total_rounds": 6
  },
  "by_severity": {
    "critical": {"found": 4, "fixed": 4, "wontfix": 0, "remaining": 0},
    "high": {"found": 18, "fixed": 18, "wontfix": 0, "remaining": 0}
  },
  "by_category": {
    "completeness": 28, "ambiguity": 19
  }
}
```

### 10.4 Report Logic

1. Scan each file as one round (in order provided, or alphabetical by filename if directory).
2. Finding identity across rounds uses `(category, location, finding_prefix)` key — same as merge deduplication (section 5.2 item 4). This ensures two different findings at the same location are tracked independently.
3. Each round: count `in_file` (total findings in file), compute `new` (key not in the UNION of ALL previous rounds' key sets), compute `resolved` (key in the immediately preceding round's key set but not in current round). A finding can only be counted as "resolved" once — when it first disappears. If a finding reappears after absence, it is counted as "new" again.
4. Convergence: `converged = true` when last 2+ rounds have 0 findings in file.
5. When findings have `resolved` field (Feature B), severity breakdown uses resolution status (fixed/wontfix/duplicate). Duplicates are counted separately: `{"found": 4, "fixed": 2, "wontfix": 1, "duplicate": 1, "remaining": 0}`. When findings lack `resolved` field, "remaining" = findings still present in latest round, "fixed" = findings that disappeared between rounds.

### 10.5 Validation

| Condition | Exit Code | Message |
|-----------|-----------|---------|
| No input files | 2 | "at least 1 findings file required for report" |
| Directory has no .ndjson files | 2 | "no .ndjson files found in {dir}" |

## 11. Feature G: Spec Reference Mode (`--spec-ref`)

### 11.1 Command

```bash
tp review spec.md --spec-ref
```

### 11.2 Behavior

Instead of embedding the full spec content inline in each prompt, the prompt says:

```
Spec file: spec.md (847 lines, 23 sections)

Read the spec file before reviewing. The spec is NOT included inline to save context.
Focus your review on:
[structured elements list]
[findings summary if applicable]
```

The `reviewResult` JSON output includes `spec_path` field with the absolute path to the spec file, so agents can programmatically access the file without parsing prompt text.

### 11.3 When To Use

- Large specs (>500 lines) where inline repetition wastes tokens.
- Agents that can read files (Claude Code, Cursor, etc.).
- NOT suitable for agents without file access.

### 11.4 Token Savings

- 800-line spec ≈ 20K chars per prompt × 3 prompts = 60K chars
- Spec-ref: ~200 chars per prompt × 3 = 600 chars
- **~99% reduction in spec content tokens**

The agent pays the cost of reading the spec once (20K) vs 3× inline (60K). Net savings: ~40K chars per round.

### 11.5 Output Changes

- `reviewResult` gets new fields: `spec_ref: true` (boolean) and `spec_path: "/absolute/path/to/spec.md"` (string).
- `review_loop.instruction` includes "Read the spec at {path} before processing each prompt."
- Structured elements summary is still inline (headings, table counts, list counts) — this is compact and helps the agent navigate.

### 11.6 Interaction with `--diff-from`

When both `--spec-ref` and `--diff-from` are active, `--spec-ref` is effectively redundant: `--diff-from` already controls what content is inline (changed sections) vs omitted (unchanged sections listed as headings). The combination is allowed but `--spec-ref` adds no additional behavior beyond what `--diff-from` already provides.

## 12. Feature Interactions

| Combination | Behavior | Notes |
|-------------|----------|-------|
| A + B | `--merge` then `--resolve` on merged file (sequential, not same invocation) | Natural workflow: merge first, then resolve |
| B + C | Resolved findings → compact summary | Core interaction — C reads B's `resolved` field |
| D + G | Diff-based + spec-ref | `--spec-ref` is redundant when `--diff-from` is active (see 11.6) |
| E + B | Verify checks resolution evidence | Verifier uses resolve data to confirm fixes |
| E + D | Verify with diff-based spec | `--verify --diff-from` uses diff content in verify prompt |
| E + G | Verify with spec-ref | `--verify --spec-ref` omits spec inline, tells agent to read file |
| F + B | Report shows resolved counts | Report uses `resolved` field for severity breakdown |

**Mutually exclusive modes** (see section 4): --merge, --resolve, --verify, --report cannot be combined with each other. All other combinations not listed above are orthogonal and require no special handling.

## 13. Implementation Order and File Organization

Tasks should be implemented in this order due to dependencies:

1. **Feature A (merge)** — no dependencies, standalone utility → `internal/cli/review_merge.go`
2. **Feature B (resolve)** — no dependencies, modifies findings format → `internal/cli/review_resolve.go`
3. **Feature C (compact summary)** — depends on B (reads `resolved` field) → modify `buildFindingsSummary` in `review.go`
4. **Feature F (report)** — depends on B (reads `resolved` field for breakdown) → `internal/cli/review_report.go`
5. **Feature G (spec-ref)** — no dependencies, modifies prompt generation → modify prompt generators in `review.go`
6. **Feature D (diff-based)** — depends on ParseHeadings, must integrate with G → `internal/engine/diff.go` + prompt changes in `review.go`
7. **Feature E (verify)** — depends on B (reads resolution evidence), C (compact summary), integrates with D and G → `internal/cli/review_verify.go`

Note: review.go is already substantial. New features A, B, D (engine), E, F get their own files to keep review.go manageable.

## 15. Test Cases

### Feature A (merge)
1. Merge 2 files with overlapping findings → dedup count correct, highest severity kept
