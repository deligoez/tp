# 0.19.0 — Agent Friction Reduction

Addresses real-world agent feedback from large-scale tp usage (100+ task projects).
Every change targets a concrete pain point reported by agents.

## Motivation

Agents using tp on projects with 100-200+ tasks reported these friction points:
- 192 manual `source_lines` fixes ("72" → "72-72")
- 5 separate batch files needed due to dependency ordering
- 21 review rounds partly caused by undetected duplicate lines and numbering gaps
- Import failures on bare JSON arrays requiring workaround via `tp init` + `tp add --bulk`
- Review agents checking code instead of spec (no disclaimer)
- `covered_by` typos with no recovery suggestion

## 1. source_lines Auto-Normalize

### 1.1 Single Number to Range

`ParseLineRanges` currently rejects single numbers (e.g., `"72"`).
Normalize them automatically to range format (`"72-72"`).

| Input | Current Behavior | New Behavior |
|-------|-----------------|--------------|
| `"72"` | Error: invalid line range | Normalized to `"72-72"` |
| `"72-72"` | OK | OK (unchanged) |
| `"4-10,72,80-90"` | Error: invalid line range: 72 | Normalized to `"4-10,72-72,80-90"` |
| `""` | OK (empty) | OK (unchanged) |

### 1.2 Normalization Point

Normalize in `ParseLineRanges()` at parse time — not at import time.
This ensures all code paths benefit: import, validate, add, add --bulk.

### 1.3 Test Cases

1. Single number `"72"` → parses as range 72-72
2. Mixed `"4-10,72,80-90"` → parses as three ranges including 72-72
3. Multiple singles `"5,10,15"` → parses as 5-5, 10-10, 15-15
4. Existing ranges unchanged: `"4-10"`, `"4-10,15-20"`
5. Empty string still returns nil

## 2. Dependency-Aware Batch Close

### 2.1 Problem

`tp done --batch` processes entries in file order. When task B depends on task A,
and both are in the same batch, B is rejected as "blocked by A" even though A
is being closed in the same batch. Agent must create multiple batch files in
dependency order — reported as creating 5 separate files for 127 tasks.

### 2.2 Solution: Topological Sort Before Processing

Before processing batch entries, sort them by dependency order:

1. Build a dependency graph from only the IDs present in the batch
2. Topological sort the batch entries
3. Process in sorted order
4. Entries not in the graph (no deps among batch entries) retain original order

### 2.3 Behavior

| Scenario | Current | New |
|----------|---------|-----|
| A→B in batch, A before B | A closes, B closes | Same |
| A→B in batch, B before A | B fails "blocked by A" | A closes first, then B |
| A→B, A not in batch, A still open | B fails | B fails (correct — A not in batch) |
| Circular dep in batch | N/A (validated earlier) | N/A |

### 2.4 Output Change

Add `reordered` field to batch output when entries were reordered:

```json
{
  "closed": 127,
  "failed": 0,
  "reordered": true,
  "results": [...]
}
```

### 2.5 Test Cases

1. 3 tasks A→B→C in reverse order in batch → all close successfully
2. 3 tasks where A depends on external task X (not in batch, still open) → A fails, B and C close if independent
3. Batch with no deps → original order preserved, `reordered` is false
4. Batch with already-done tasks → skipped (idempotent), deps satisfied
5. Mixed: some need reorder, some independent → correct interleaving

## 3. Spec Lint: Duplicate Consecutive Lines

### 3.1 Rule

Detect two or more consecutive identical non-empty lines in spec.
These are edit artifacts that waste review agent attention.

### 3.2 Finding Format

```json
{
  "line": 45,
  "severity": "warning",
  "rule": "duplicate-line",
  "message": "duplicate consecutive line",
  "context": "The duplicated text..."
}
```

### 3.3 Scope

- Only non-empty lines (ignore blank line runs)
- Only consecutive (not scattered duplicates)
- Applies to spec linting (`tp lint`), not task file validation

### 3.4 Test Cases

1. Two identical consecutive lines → 1 warning at second line
2. Three identical consecutive lines → 2 warnings
3. Identical lines separated by blank line → no warning
4. Empty consecutive lines → no warning
5. Code block contents excluded (between ``` fences)

## 4. Spec Lint: Section Numbering Gaps

### 4.1 Rule

Detect gaps in numbered section headings. E.g., `### 4.2` followed by `### 4.4` (missing 4.3).

### 4.2 Finding Format

```json
{
  "line": 88,
  "severity": "warning",
  "rule": "numbering-gap",
  "message": "section numbering gap: ### 4.2 → ### 4.4 (missing 4.3)",
  "context": "### 4.4 Missing Section"
}
```

### 4.3 Detection Logic

1. Parse headings with numeric prefixes (e.g., `## 2.`, `### 2.1`, `#### 2.1.3`)
2. Group by parent prefix and heading level
3. Within each group, check that numbers increment by 1
4. Report gaps with the expected missing number

### 4.4 Scope

- Only headings with numeric prefixes
- Non-numeric headings (e.g., `## Overview`) are ignored
- Applies to spec linting (`tp lint`)

### 4.5 Test Cases

1. `### 4.1`, `### 4.2`, `### 4.3` → no warning
2. `### 4.1`, `### 4.3` → warning: missing 4.2
3. `### 4.1`, `### 4.5` → warning: missing 4.2 (reports first gap)
4. Mixed levels: `## 2.`, `### 2.1`, `### 2.3` → warning at 2.3
5. Non-numeric headings interspersed → ignored, no false positives

## 5. Import: Accept Bare JSON Array

### 5.1 Problem

`tp import` only accepts full TaskFile format. Agents often produce bare JSON arrays of tasks,
requiring workaround: `tp init` + `tp add --bulk`.

### 5.2 Solution

When `tp import` detects a JSON array at top level:
1. Wrap it into a TaskFile structure
2. Auto-detect spec path from existing `.tasks.json` or prompt with error
3. Set version=1, created_at=now, empty workflow/coverage
4. Proceed with normal import validation

### 5.3 Detection

```go
// Peek at first non-whitespace byte
if firstByte == '[' {
    // Parse as []Task, wrap into TaskFile
} else {
    // Parse as TaskFile (existing behavior)
}
```

### 5.4 Requirement: --spec Flag

When importing a bare array, `--spec <path>` is required (no way to infer it):

```bash
tp import tasks.json --spec spec/feature.md    # Bare array + spec path
tp import taskfile.json                         # Full TaskFile (spec embedded)
```

If bare array detected and `--spec` not provided, error with hint:
```
error: bare JSON array detected; use --spec <path> to specify the spec file
hint: or wrap in TaskFile format: {"version":1,"spec":"...","tasks":[...]}
```

### 5.5 Test Cases

1. Full TaskFile JSON → works as before
2. Bare JSON array + `--spec` → wraps and imports
3. Bare JSON array without `--spec` → error with hint
4. Bare array with invalid tasks → validation errors (same as TaskFile)
5. NDJSON input (lines starting with `{`) → error: "use tp add --bulk for NDJSON"

## 6. Review Prompt: Spec-Only Disclaimer

### 6.1 Problem

Review agents sometimes check implementation code and report "not implemented" findings.
This wastes rounds — review is about spec quality, not implementation status.

### 6.2 Solution

Add disclaimer to all review prompts that do NOT use `--affected-files` or `--perspective code-audit`:

```
IMPORTANT: This is a SPEC REVIEW. Review ONLY the spec document text.
Do NOT check implementation code or report "not implemented" findings.
Focus on: completeness, ambiguity, contradictions, missing edge cases, testability.
```

### 6.3 Placement

- Prepend to each generated prompt's instruction section
- Only when `--affected-files` is NOT used and perspective is NOT `code-audit`
- When `--affected-files` IS used, the existing code-aware instructions apply instead

### 6.4 Test Cases

1. Default review → disclaimer present in all 3 prompts
2. `--affected-files` review → no disclaimer (code-aware mode)
3. `--perspective code-audit` → no disclaimer (code audit mode)
4. `--perspective documentation` → disclaimer present
5. `--perspective testing` → disclaimer present

## 7. Covered-by: Did-You-Mean Suggestion

### 7.1 Problem

When `covered_by` references a non-existent task ID, the error gives no recovery path.
Agent must manually find the correct ID.

### 7.2 Solution

When covered_by ID is not found, compute string similarity against all task IDs
and suggest the closest match(es):

```json
{
  "id": "linker-detect",
  "error": "covered_by: task \"linker-detect-strategy\" not found",
  "hint": "did you mean: linker-detect-impl, linker-strategy?"
}
```

### 7.3 Algorithm

Simple prefix + substring match (no need for Levenshtein):
1. Check prefix match (ID starts with the given string or vice versa)
2. Check if any word in the given ID appears in existing IDs
3. Return top 3 matches, sorted by relevance
4. If no matches, hint: "use tp list --ids to see all task IDs"

### 7.4 Scope

- `tp done --covered-by` (single)
- `tp done --batch` with `covered_by` field
- `tp close` does not support covered_by (only tp done does)

### 7.5 Test Cases

1. Exact match → no error (existing behavior)
2. Prefix match: "linker-detect-strategy" vs "linker-detect-impl" → suggests "linker-detect-impl"
3. Substring match: "auth-model" vs "user-auth-model" → suggests "user-auth-model"
4. No match at all → "use tp list --ids to see all task IDs"
5. Multiple matches → top 3 shown
