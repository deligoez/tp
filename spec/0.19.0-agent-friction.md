# 0.19.0 — Agent Friction Reduction

Addresses real-world agent feedback from large-scale tp usage (100+ task projects).
Every change targets a concrete pain point reported by agents.

## Motivation

Agents using tp on projects with 100-200+ tasks reported these friction points:
- 192 manual `source_lines` fixes ("72" → "72-72")
- 5 separate batch files needed due to dependency ordering
- 21 review rounds partly caused by undetected duplicate lines and numbering gaps
- Import failures on bare JSON arrays requiring workaround via `tp init` + `tp add --bulk`
- Review agents checking code instead of spec (no disclaimer)
- `covered_by` typos with no recovery suggestion

## Backward Compatibility

All changes in this spec are additive or relaxations of existing strictness:

| Change | Type | Impact |
|--------|------|--------|
| source_lines normalize | Relaxation | Previously-invalid input now accepted; no existing valid input changes behavior |
| Batch toposort | Additive | New `reordered` field in output; existing ordered batches behave identically |
| Duplicate line lint | Additive | New lint rule; no existing rules change |
| Numbering gap lint | Additive | New lint rule; no existing rules change |
| Import bare array | Additive | New input format accepted; existing TaskFile format unchanged |
| Review disclaimer | Additive | New text prepended to prompts; no existing prompt text removed |
| covered_by suggestion | Additive | New `hint` field on error; existing error message unchanged |

## 1. source_lines Auto-Normalize

### 1.1 Single Number to Range

`ParseLineRanges` currently rejects single numbers (e.g., `"72"`).
Normalize them automatically to range format (`"72-72"`).

| Input | Current Behavior | New Behavior |
|-------|-----------------|--------------:|
| `"72"` | Error: invalid line range | Normalized to `"72-72"` |
| `"72-72"` | OK | OK (unchanged) |
| `"4-10,72,80-90"` | Error: invalid line range: 72 | Normalized to `"4-10,72-72,80-90"` |
| `""` | OK (empty) | OK (unchanged) |

**Existing error behavior is preserved** for truly invalid inputs: negative numbers (`"-5"`),
zero (`"0"`), non-numeric strings (`"abc"`), reversed ranges (`"10-5"`), and partial ranges (`"72-"`).
These continue to return errors as before.

### 1.2 Normalization Point

Normalize in `ParseLineRanges()` at parse time — not at import time.
This ensures all code paths benefit: import, validate, add, add --bulk.

### 1.3 Test Cases

1. Single number `"72"` → parses as range 72-72
2. Mixed `"4-10,72,80-90"` → parses as three ranges including 72-72
3. Multiple singles `"5,10,15"` → parses as 5-5, 10-10, 15-15
4. Existing ranges unchanged: `"4-10"`, `"4-10,15-20"`
5. Empty string still returns nil
6. Invalid inputs still error: `"-5"`, `"abc"`, `"10-5"`, `"72-"`

## 2. Dependency-Aware Batch Close

### 2.1 Problem

`tp done --batch` processes entries in file order. When task B depends on task A,
and both are in the same batch, B is rejected as "blocked by A" even though A
is being closed in the same batch. Agent must create multiple batch files in
dependency order — reported as creating 5 separate files for 127 tasks.

### 2.2 Solution: Topological Sort Before Processing

Before processing batch entries, sort them by dependency order:

1. Build a dependency graph from only the IDs present in the batch
2. Include `covered_by` references in the graph (covered task must close before covering task)
3. Remove already-done tasks from the graph (they are resolved as `skipped` before cycle detection)
4. If a cycle is detected among remaining batch entries, fail all cycle-member entries with error and hint: `"dependency cycle: A → B → A"`, hint: `"break the cycle in the task file"`; non-cycle entries still process
5. Topological sort using stable sort — among entries at the same topological depth, original file order is preserved
6. For each entry in sorted order: if task is already done, record as `skipped` and treat it as a satisfied dependency for subsequent batch entries that depend on it
7. Unknown task IDs (not in task file) are detected during processing and reported as failures, same as current behavior

### 2.3 Behavior

| Scenario | Current | New |
|----------|---------|-----|
| A→B in batch, A before B | A closes, B closes | Same |
| A→B in batch, B before A | B fails "blocked by A" | A closes first, then B |
| Diamond (A→B→D, A→C→D) all in batch | D fails | All close in topological order |
| A→B, A not in batch, A still open | B fails | B fails with hint: "close A first (not in batch)" |
| A→B, A not in batch, A already done | B succeeds | Same (A's dep already satisfied) |
| A→B, A not in batch, A is wip | B fails | B fails with hint: "A is wip, not done" |
| A already done, in batch | N/A | Skipped (idempotent), deps satisfied for dependents |
| Duplicate ID in batch | Second entry gets confusing error | Second entry skipped (already closed by first) |
| Circular dep among batch entries | N/A | Cycle members fail with hint: "dependency cycle: A → B → A"; others process |

### 2.4 Output Change

`reordered` field is **always present** in batch output (true when entries were reordered, false otherwise).
`skipped` count tracks already-done entries. Skipped entries appear in results with `"status":"skipped"`.
Results array entry schema:

```json
{"id":"task-id", "status":"closed|failed|skipped", "error":"...", "hint":"..."}
```
```json
{
  "closed": 125,
  "failed": 0,
  "skipped": 2,
  "reordered": true,
  "results": [...]
}
```

### 2.5 Test Cases

1. 3 tasks A→B→C in reverse order in batch → all close successfully, `reordered: true`
2. 3 tasks where A depends on external task X (not in batch, still open) → A fails with hint "close X first (not in batch)", B and C close if independent
3. Batch with no deps → original order preserved, `reordered: false`
4. Batch with already-done tasks → `status: "skipped"` in results, skipped count incremented, deps satisfied for dependents
5. Batch [C, B, A] where B depends on A, C is independent → processed as [C, A, B] (same-depth entries preserve original file order; C at depth 0, A at depth 0, B at depth 1)
6. Duplicate ID in batch → second entry skipped
7. covered_by reference in batch → referenced task processed first
8. Two independent tasks [X, Y] with no deps → order preserved as [X, Y]

## 3. Spec Lint: Duplicate Consecutive Lines

### 3.1 Rule

Detect two or more consecutive identical non-empty lines in spec.
These are edit artifacts that waste review agent attention.

### 3.2 Finding Format

```json
{
  "line": 45,
  "severity": "warning",
  "rule": "duplicate-line",
  "message": "duplicate consecutive line",
  "context": "The duplicated text (truncated to 80 chars)..."
}
```

The `context` field is truncated to 80 characters.

### 3.3 Scope

- A line is "empty" if it is zero-length or contains only whitespace (spaces, tabs)
- Only non-empty lines trigger duplicate detection (consecutive blank lines are ignored)
- Only consecutive duplicates (not scattered)
- Fenced code block contents excluded — opening fence is a line starting with three or more backticks, optionally followed by a language identifier (e.g., ` ```json `); indented code blocks and inline code are still checked
- Applies to spec linting (`tp lint`), not task file validation

### 3.4 Test Cases

1. Two identical consecutive lines → 1 warning at second line
2. Three identical consecutive lines → 2 warnings (at lines 2 and 3)
3. Identical lines separated by blank line → no warning
4. Empty consecutive lines → no warning
5. Code block contents excluded (between ``` fences)

## 4. Spec Lint: Section Numbering Gaps

### 4.1 Rule

Detect gaps in numbered section headings. E.g., `### 4.2` followed by `### 4.4` (missing 4.3).

### 4.2 Finding Format

```json
{
  "line": 88,
  "severity": "warning",
  "rule": "numbering-gap",
  "message": "section numbering gap: ### 4.2 → ### 4.4 (missing 4.3)",
  "context": "### 4.4 Missing Section"
}
```

### 4.3 Detection Logic

1. Parse headings with numeric prefixes (e.g., `## 2.`, `### 2.1`, `#### 2.1.3`)
2. Group by parent prefix and heading level
3. The first heading in each group establishes the starting number; only subsequent gaps within the group are reported (e.g., if a group starts at `### 4.3`, only gaps after 4.3 are reported — 4.1 and 4.2 are not flagged)
4. Report one warning per missing number (e.g., gap from 4.1 to 4.5 → three warnings for 4.2, 4.3, 4.4)

### 4.4 Scope

- Only headings with numeric prefixes
- Non-numeric headings (e.g., `## Overview`) are ignored
- Applies to spec linting (`tp lint`)

### 4.5 Test Cases

1. `### 4.1`, `### 4.2`, `### 4.3` → no warning
2. `### 4.1`, `### 4.3` → 1 warning: missing 4.2
3. `### 4.1`, `### 4.5` → 3 warnings: missing 4.2, 4.3, 4.4
4. Mixed levels: `## 2.`, `### 2.1`, `### 2.3` → 1 warning at 2.3 (missing 2.2)
5. Non-numeric headings interspersed → ignored, no false positives

## 5. Import: Accept Bare JSON Array

### 5.1 Problem

`tp import` only accepts full TaskFile format. Agents often produce bare JSON arrays of tasks,
requiring workaround: `tp init` + `tp add --bulk`.

### 5.2 Solution

When `tp import` detects a JSON array at top level:
1. Wrap it into a TaskFile structure
2. Require `--spec <path>` flag (see Section 5.4)
3. Set version=1, created_at=now, empty workflow/coverage
4. Proceed with normal import validation
5. Follow same `--force` semantics as full TaskFile import (without --force, error if task file already exists)

### 5.3 Detection

```go
// Peek at first non-whitespace byte
if firstByte == '[' {
    // Parse as []Task, wrap into TaskFile
} else if firstByte == '{' {
    // Parse as TaskFile (existing behavior)
    // If TaskFile parse fails AND file has 2+ lines starting with '{',
    // include BOTH the original parse error AND the NDJSON suggestion:
    // error: "invalid task file: <parse error>; if this is NDJSON, use tp add --bulk"
} else {
    // Error: invalid JSON
}
```

### 5.4 Requirement: --spec Flag

When importing a bare array, `--spec <path>` is required (no way to infer it from a bare array):

```bash
tp import tasks.json --spec spec/feature.md    # Bare array + spec path
tp import taskfile.json                         # Full TaskFile (spec embedded)
```

If bare array detected and `--spec` not provided, error with hint:
```
error: bare JSON array detected; use --spec <path> to specify the spec file
hint: or wrap in TaskFile format: {"version":1,"spec":"...","tasks":[...]}
```

### 5.5 Test Cases

1. Full TaskFile JSON → works as before
2. Bare JSON array + `--spec` → wraps and imports
3. Bare JSON array without `--spec` → error with hint
4. Bare array with invalid tasks → validation errors (same as TaskFile)
5. NDJSON input (multiple lines starting with `{`, TaskFile parse fails) → error: "use tp add --bulk for NDJSON"

## 6. Review Prompt: Spec-Only Disclaimer

### 6.1 Problem

Review agents sometimes check implementation code and report "not implemented" findings.
This wastes rounds — review is about spec quality, not implementation status.

### 6.2 Solution

Add disclaimer to review prompts in spec-only modes:

```
IMPORTANT: This is a SPEC REVIEW. Review ONLY the spec document text.
Do NOT check implementation code or report "not implemented" findings.
Focus on: completeness, ambiguity, contradictions, missing edge cases, testability.
```

### 6.3 Placement

Prepend to each generated prompt's instruction section. Mode applicability:

| Mode | Disclaimer |
|------|-----------|
| Default review (`tp review spec.md`) | Yes |
| `--perspective documentation` | Yes |
| `--perspective testing` | Yes |
| `--diff-from` (diff-based review) | Yes |
| `--spec-inline` (full spec inline) | Yes |
| `--affected-files` (code-aware) | No (code-aware instructions apply instead) |
| `--perspective code-audit` | No (code audit instructions apply instead) |
| `--verify` (verification mode) | No (verifier has its own instructions) |

### 6.4 Test Cases

1. Default review → disclaimer present in all 3 prompts
2. `--affected-files` review → no disclaimer (code-aware mode)
3. `--perspective code-audit` → no disclaimer (code audit mode)
4. `--perspective documentation` → disclaimer present
5. `--verify` mode → no disclaimer (verifier role has own instructions)

## 7. Covered-by: Did-You-Mean Suggestion

### 7.1 Problem

When `covered_by` references a non-existent task ID, the error gives no recovery path.
Agent must manually find the correct ID.

### 7.2 Solution

When covered_by ID is not found, compute string similarity against all task IDs
and suggest the closest match(es):

```json
{
  "id": "linker-detect",
  "error": "covered_by: task \"linker-detect-strategy\" not found",
  "hint": "did you mean: linker-detect-impl, linker-strategy?"
}
```

### 7.3 Algorithm

Simple prefix + substring match (no Levenshtein — character transposition typos are out of scope):

1. Split the given ID and all existing IDs into words on hyphens (`-`)
2. Check prefix match: given ID starts with an existing ID or vice versa
3. Check word overlap: any word (3+ characters) from the given ID appears in an existing ID
4. Rank matches: prefix match > word overlap; within each tier, shorter IDs first
5. Return top 3 matches
6. If no matches found, hint: "use tp list --ids to see all task IDs"

### 7.4 Scope

This feature applies only to `tp done` (the `--covered-by` flag and `covered_by` field in batch NDJSON).
`tp close` has no covered_by support and is unaffected.

### 7.5 Test Cases

1. Exact match → no error (existing behavior)
2. Prefix match: "linker-detect-strategy" vs "linker-detect-impl" → suggests "linker-detect-impl"
3. Word overlap: "auth-model" vs "user-auth-model" → suggests "user-auth-model"
4. No match at all → "use tp list --ids to see all task IDs"
5. Multiple matches → top 3 shown, prefix matches ranked first
