# v0.25.0 — Reviewer/Auditor Prompt-Design Research Notes

Non-normative process artifact (§14.4). This is the recorded web-research pass on
reviewer/auditor prompt design, performed **before** the embedded default corpus
(`default-corpus-data`) and the §14 SKILL authoring guidance
(`skill-authoring-guidance`) are written, so both are informed by current
practice. Findings are organized by the four §14.2 dimensions.

## 1. Adversarial framing — "try to refute", not "check whether fine"

- LLM self-review has a **systematic leniency bias**: a reviewer that shares the
  builder's blind spots fails in correlated ways and rubber-stamps. The mitigation
  is to frame the role as adversarial *by design* — explicitly rigorous and
  skeptical, told to hunt for flaws (factual errors, logical gaps, missing
  requirements), counterbalancing the generator's helpfulness bias.
- A reviewer with **conflicting goals performs poorly**; a role whose sole job is
  to break the artifact outperforms one asked to both judge and be agreeable.
- LLMs **underweight negation**, so a spec's "DO NOT" constraints are the weakest
  link in probabilistic review — a role should be told to actively enumerate and
  test negative/forbidden constraints rather than assume they hold.
- Practical wording: "find where this breaks", "assume it is wrong and prove it",
  "enumerate every X and verify each" beat "confirm this looks correct".

## 2. Evidence demand — every finding cites a location and a why

- **Requiring a citation for every claim creates a verifiable chain** and
  constrains output to a known source, cutting hallucination. A finding with no
  location is unverifiable and un-clusterable.
- Two hallucination modes to guard against: **factuality** (contradicts the world)
  and **faithfulness** (contradicts its own source). Demanding a `location` anchor
  plus a stated reason lets a downstream verifier check faithfulness cheaply —
  this is exactly what tp's §8.2 section-anchor `location` and the audit
  PASS/PARTIAL/FAIL `status` make meaningful.
- A finding without evidence should be treated as not-yet-a-finding: the prompt
  should demand "quote or cite the exact spec location and explain why it fails".

## 3. Scope boundaries — state what the role does NOT cover

- Running many parallel agents **without strong role differentiation yields
  overlapping outputs** that are expensive to synthesize (and, in tp, get flagged
  as trim candidates by §8.5). An explicit "you do NOT cover A, B, C" clause keeps
  lenses disjoint.
- Scope boundaries also reduce off-target noise: a security role told "ignore
  style and naming" spends its budget on trust boundaries, not bikeshedding.
- Boundaries should name the *sibling* roles' territory so the panel tiles the
  space without gaps or double-coverage.

## 4. One-lens-per-role — a role targets one distinct failure mode

- **Perspective diversity produces insight a single prompt misses**; role-based
  multi-agent panels improve specificity and diversity of findings, and
  *increasing the diversity of roles* (not their count) improves results.
- Distinct roles ask *legitimately different questions about the same artifact*
  (e.g., correctness vs. security vs. contract/performance). A role that overlaps
  another's question is redundant — cost with no marginal signal.
- Each role should emit **structured findings** (the fixed tp output contract:
  role, location, class, severity [+ status for audit]), which makes the
  synthesis/merge and the per-role overlap report clean.

## Synthesis for corpus authoring (feeds `default-corpus-data`)

Each embedded default role should exemplify all four: an adversarial system-prompt
body ("try to break …"), a `focus` list that demands evidence/locations, an
explicit non-coverage boundary, and a single failure-lens distinct from its
siblings. Concretely:

- **software reviewers** — implementer (does the described change actually work:
  error paths, edge cases, happy-path gaps), tester (is every claim verifiable /
  falsifiable, are acceptance criteria testable), architect (backward-compat,
  performance, cross-cutting/interface consistency). These three lenses genuinely
  diverge.
- **software auditors** — spec-coverage (is every spec element implemented),
  security (trust boundaries, injection, unsafe defaults), maintainability-
  conventions (does the code match project conventions/idioms).
- **prose reviewers** — coherence (narrative/structural continuity, does one part
  contradict or pre-empt another) and soundness (expository derivability — can each
  claim be derived from what precedes it without inventing facts). Prose defaults
  to the leaner 2-reviewer panel (§6.3) because prose flaws surface from many
  angles at once, so a third reviewer mostly overlaps.
- **prose auditors** — spec-coverage and soundness.

Other domains and their characteristic diverging lenses (for the §14.3 catalog):
legal/contract (obligation completeness vs. ambiguity/loophole), product/PRD
(user-journey completeness vs. measurable-acceptance), data-schema (referential
integrity vs. migration/compat), academic (claim support vs. methodology).
