# 0.22.0 — Section Name Normalization

Resolve the long-standing inconsistency between `tp lint` output and `tp import` / `tp validate` expectations for `source_sections` values, plus document `source_sections` format and `context_only` semantics in SKILL.md.

## Motivation

Real agent feedback (project consuming tp v0.21.0):

> `tp lint --json` and `tp import` are inconsistent on section name normalization. `tp lint` returns
> `structured_elements.numbered_lists[].heading` as raw heading text like `"4. Backend Migration"`,
> but `tp import` rejects the same string with a non-existent-section coverage error.
> The skill docs don't mention `source_sections` expected format or `context_only` usage.

This is a textbook AX (Agent Experience) miss:
- Agent reads heading from `tp lint --json`
- Agent uses identical string in task `source_sections`
- `tp import` rejects with no actionable hint
- Wasted round-trip; no learning signal in the error

Root cause: `tp lint` reports `parseHeadingLine()` text only (no `#` prefix), while
`coverage.go` and `validate.go` build the lookup table as `prefix + " " + text`
(e.g. `"## 4. Backend Migration"`). Two normalization domains; no bridge between them.

## Goals

1. **Lenient acceptance**: `tp import` accepts both `"4. Backend Migration"` (from lint) and
   `"## 4. Backend Migration"` (canonical). No agent retraining required.
2. **Auto-normalize on import**: persisted task file always stores canonical form
   (`"## 4. Backend Migration"`), so downstream tools see one format.
3. **Ambiguity guard**: when `tp import` cannot uniquely resolve a prefix-less name
   (e.g. spec has both `## Setup` and `### Setup`), reject with actionable error.
4. **Did-you-mean error messages**: when a `source_sections` value cannot be resolved,
   show expected format + closest 3 candidates (reuse `engine/suggest.go` Levenshtein).
5. **Documentation**: SKILL.md gains `source_sections` format example + `context_only` semantics.
6. **Non-breaking**: `tp lint --json` output schema unchanged (still emits raw text in `heading`).

## Non-Goals

- Changing `tp lint` JSON output schema (would break downstream parsers).
- Auto-generating `source_sections` from spec scanning (separate feature).
- Renaming `context_only` field (cosmetic; field name kept stable for compatibility).

## 1. Section Resolution Helper

### 1.1 New function ResolveSection

Add a single helper used by both coverage and validation paths.

```go
// ResolveSection attempts to resolve a user-supplied section name to a canonical
// "## prefix + text" heading from the spec. Returns:
//   resolved   — canonical form ("## 4. Backend Migration") if a unique match found
//   ambiguous  — true if multiple headings share the same text but different levels
//   candidates — full list of canonical matches when ambiguous (for error messages)
func ResolveSection(s string, headings []*Heading) (resolved string, ambiguous bool, candidates []string)
```

### 1.2 Resolution rules

| Input | Resolution |
|-------|------------|
| `"## 4. Backend Migration"` (canonical) | Returns `"## 4. Backend Migration", false, nil` if heading exists; `"", false, nil` otherwise |
| `"4. Backend Migration"` (text only) | Returns canonical form if exactly one heading text matches; `"", true, [candidates]` if multiple levels match; `"", false, nil` if no match |
| Whitespace variations (`"##  Setup"`, `"## Setup "`) | Trimmed then resolved as canonical |
| Empty string | Returns `"", false, nil` (no match) |

### 1.3 Example resolution table

Given spec headings: `## Setup`, `### Setup`, `## 4. Backend Migration`

| Input | Resolved | Ambiguous | Candidates |
|-------|----------|-----------|------------|
| `## 4. Backend Migration` | `## 4. Backend Migration` | false | nil |
| `4. Backend Migration` | `## 4. Backend Migration` | false | nil |
| `## Setup` | `## Setup` | false | nil |
| `Setup` | `""` | true | `["## Setup", "### Setup"]` |
| `Nonexistent` | `""` | false | nil |

## 2. Import Auto-Normalize

### 2.1 Behavior

`tp import` (and `tp add`) MUST normalize each task's `source_sections` to canonical form
before persisting to the task file:

1. For each task, for each entry in `source_sections`:
   - Call `ResolveSection(entry, specHeadings)`.
   - If `resolved != ""`: replace entry with canonical form.
   - If `ambiguous`: abort import with error listing candidates.
   - If neither: leave entry unchanged (validation will catch it with did-you-mean).
2. After normalization, run existing `AutoFillCoverage` and validation.

### 2.2 Error behavior

| Scenario | Behavior |
|----------|----------|
| All entries resolve cleanly | Import succeeds, task file holds canonical form |
| Entry ambiguous (level conflict) | Import fails with exit code 1, error names task ID + ambiguous entry + candidates |
| Entry unresolvable (typo / missing) | Import succeeds (current behavior), but validation reports it via did-you-mean |

Rationale: ambiguity is a structural error the agent cannot recover from without guidance;
unresolvable entries are caught by `tp validate` with full context (existing flow).

### 2.3 tp add parity

`tp add` (single task add) MUST apply the same normalization. This preserves the
P2 (Batch Parity) principle — single-task path matches batch import path.

## 3. Improved Coverage Error Messages

### 3.1 Current message

```
task X references non-existent section: 4. Backend Migration
```

### 3.2 New message

```
task X: source_sections entry "4. Backend Migration" not found in spec.
Expected canonical format: "## 4. Backend Migration" (heading marker prefix + text).
Did you mean:
  - ## 4. Backend Migration
  - ## 4. Frontend Migration
  - ### 4.1 Backend Schema
```

### 3.3 Suggestion algorithm

Reuse `engine/suggest.go` Levenshtein-based suggestion:
- Compare input against full canonical heading list.
- Return up to 3 closest matches (max distance: half of input length).
- If zero suggestions found, omit the "Did you mean" block.

### 3.4 Ambiguous case message

```
task X: source_sections entry "Setup" is ambiguous — multiple headings match:
  - ## Setup
  - ### Setup
Use the full canonical form (e.g. "## Setup") to disambiguate.
```

## 4. Documentation Updates

### 4.1 SKILL.md additions

A new subsection under "Decomposition Rules" or near `source_sections` mention:

```markdown
### source_sections format

Each entry MUST match a heading in the spec, in canonical form:
`"## Heading Text"` (heading marker prefix + space + heading text).

Example: spec contains `## 4. Backend Migration` → use `"## 4. Backend Migration"` in source_sections.

tp import is lenient — `"4. Backend Migration"` (without prefix) is also accepted
when unambiguous and is auto-normalized to canonical form. Use full canonical form
when the same text appears at multiple heading levels.
```

A new subsection explaining `context_only`:

```markdown
### Coverage block: context_only vs unmapped

`coverage.context_only` lists spec headings that no task references in source_sections.
These are "context-only" headings — present in the spec for reading flow, but no
implementation task is needed (intro paragraphs, motivation, examples).

`coverage.unmapped` lists spec headings that are NOT in any task's source_sections
AND are not declared as context_only. tp validate treats these as errors —
every spec heading must either be mapped to a task or explicitly context_only.

Auto-fill (during tp import) marks all unreferenced headings as context_only by default.
```

### 4.2 CLAUDE.md additions

Under "Self-Development > Rules" — add format reminder for self-development:

```markdown
- `source_sections` MUST use canonical form `"## Heading Text"` (with `##` prefix and space)
- Lenient acceptance exists for v0.22.0+ but prefer canonical form for clarity in spec files
```

### 4.3 README.md additions

Brief mention near existing `source_sections` examples:

```markdown
Section names use canonical form: `"## Heading Text"` (with the heading marker prefix).
Plain text (`"Heading Text"`) is accepted and auto-normalized when unambiguous.
```
