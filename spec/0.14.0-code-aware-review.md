# Code-Aware Review

**Status:** Draft
**Date:** 2026-04-02

## 1. Problem

`tp review` examines only the spec text. Review agents never see the actual source code that the spec is supposed to change. This causes five categories of missed findings:

1. **State-dependent code behaviors go unreviewed.** Vue template bindings (`:disabled`, `:loading`, `v-if`, `:class`) exist in code but the spec never mentions them. Review agents can't flag these gaps because they never read the code.

2. **Acceptance criteria describe removals, not final states.** Task acceptance says "isPhoneCheckInProgress computed was removed" — correct but insufficient. Should say "phone input `:disabled` only when `isFormLocked`, no other condition" — which would catch the missed second usage.

3. **Line coverage doesn't measure code element coverage.** `tp validate` checks which spec lines map to tasks. But it has no concept of code elements: "of the 15 `:disabled` bindings in AddCustomer.vue, how many appear in any task's acceptance criteria?" Zero — because tp never reads the code.

4. **Convergence is false.** Round 4: all 3 agents say "spec is solid". But none read the Vue template. Convergence criterion is "no new high-severity findings" — this means "agents didn't find more", not "agents checked everything". Thoroughness is unmeasured.

5. **Intra-file scope is implicit.** Spec says "Backend unchanged, other machines untouched" — file-level granularity. But within AddCustomer.vue, which specific references to `isIdentityCheckInProgress` are "in scope for removal" vs "accidental survivors"? Spec doesn't distinguish.

## 2. Root Cause

tp is spec-space-only. It generates prompts from spec text. It never injects source code into review context. Agents review the spec in isolation — they can't compare spec intent against code reality.

## 3. Proposed Solution

Two new mechanisms:

**A. `--affected-files` flag** — injects source file contents into any review prompt, making all perspectives code-aware.

**B. `--perspective code-audit`** — dedicated perspective that systematically compares code against spec, producing findings for every state-dependent behavior the spec misses.

Additionally, two lint improvements:

**C. Acceptance quality lint** — warns when task acceptance criteria use weak patterns (negative-only assertions, vague language).

**D. Affected files scope lint** — warns when spec's affected files table has "Modify" rows without scope descriptions.

## 4. Design Principles

| # | Principle | Application |
|---|-----------|-------------|
| P1 | AX First | `--affected-files` is a single flag. tp reads files, agent analyzes. No framework-specific parsing in tp. |
| P2 | Batch Parity | Same flag works for 1 file or 50 files. Same output shape. |
| P3 | Minimal Tokens | File contents capped per-file and total. Only affected files, not entire codebase. |
| P4 | Agent Plans, Tool Executes | tp reads files and generates prompts. Agent (AI) does the intelligent code analysis. tp doesn't parse Vue/React/templates. |

## 5. Command Changes

### 5.1 New Flags

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--affected-files` | []string | [] | Paths to source files to inject into review context |
| `--final-round` | bool | false | Force mandatory code read-through in review prompts |

### 5.2 Validation Rules

1. `--affected-files` accepts one or more paths.
2. Each path in `--affected-files` must exist. Exit code 3 if any file not found.
3. Each path in `--affected-files` must be a file (not directory). Exit code 3 if any path is a directory.
4. `--perspective code-audit` requires at least one `--affected-files`. Exit code 2 if missing.
5. `--final-round` requires `--round >= 2`. Exit code 2 if used with round 1.
6. `--final-round` without `--affected-files` produces warning: "final-round without affected-files: agents won't read code"
7. `--affected-files` is compatible with all perspectives (default, documentation, testing, code-audit).
8. `--affected-files` is compatible with `--round` and `--findings`.
9. Duplicate paths in `--affected-files` are deduplicated.

### 5.3 Behavior Matrix

| Perspective | `--affected-files` | Behavior |
|-------------|-------------------|----------|
| (default) | not provided | Existing 3-prompt behavior (unchanged) |
| (default) | provided | 3 prompts with code context injected |
| `code-audit` | provided | Single code audit prompt with code context |
| `code-audit` | not provided | Exit code 2 (required) |
| `documentation` | not provided | Doc change plan prompt, no code context |
| `documentation` | provided | Doc prompt with code files alongside doc files (enables drift detection: "code does X but docs say Y") |
| `testing` | provided | Test prompt with code context (optional, for existing test analysis) |
| any | `--final-round` | Prompts get mandatory code read-through instruction |

## 6. Affected Files Context Injection

### 6.1 Context Collection

When `--affected-files` is provided:

1. Deduplicate file paths.
2. Verify all paths exist and are files (exit 3 if not).
3. Read file contents.
4. Cap each file at 8000 chars. Truncate with `[...truncated at N chars]`.
5. Cap total affected-files content at 50000 chars. Stop reading files when cap hit.
6. When combined with `--findings`, enforce a total prompt budget of 60000 chars. Individual caps are: spec (10000), affected files (50000), findings summary (5000). When all three are present, the affected-files cap is dynamically reduced so the total stays within budget: `affected_cap = 60000 - len(spec) - len(findings_summary)`. Per-file cap remains at 8000. Document this in the prompt output as "affected files truncated to fit prompt budget".
7. Build a summary: file paths with line counts.
8. Inject into prompt as "Affected Files" section.

### 6.2 Context Format in Prompt

```
## Affected Files

Files: src/components/AddCustomer.vue (247 lines), src/api/customers.ts (89 lines)

### src/components/AddCustomer.vue
<template>
  <input :disabled="isFormLocked || isPhoneCheckInProgress" />
  ...
</template>

### src/api/customers.ts
export async function createCustomer(data) {
  ...
}
```

### 6.3 Integration with Existing Prompts

When `--affected-files` is provided with default 3-prompt review:

**Implementer prompt** — append to numbered checklist:
```
N+1. For each state-dependent behavior in the affected files (disabled, loading, visibility, conditional rendering), verify the spec addresses it. What controls each condition?
```

**Tester prompt** — append to numbered checklist:
```
N+1. For each affected file, what test verifies the current behavior changes to match the spec? What existing behavior breaks if only the spec's described changes are made?
```

**Architect prompt** — append to numbered checklist:
```
N+1. Do the affected files contain behavior not described in the spec? Could any state-dependent logic in the affected files produce unintended side effects?
```

### 6.4 Integration with Final Round

When `--final-round` is provided:

All prompts get an additional numbered checklist item appended (after the affected files item, before the finding format):
```
MANDATORY: Read every file in the Affected Files section line-by-line. For each state-dependent behavior (disabled, loading, conditional rendering, class binding, error handling), verify the spec explicitly addresses it. Do NOT report "spec is solid" unless you have verified every state-dependent element.
```

### 6.5 Integration with Perspective Prompts

When `--affected-files` is provided with `--perspective documentation`:
- Code files are injected into the doc prompt alongside doc files.
- Agent can detect drift: "code does X but docs say Y".

When `--affected-files` is provided with `--perspective testing`:
- Code files are injected alongside test files.
- Agent can see what's tested and what's not.

## 7. Code-Audit Perspective

### 7.1 Purpose

Systematically compare code against spec. Find state-dependent behaviors in code that the spec doesn't address.

### 7.2 Prompt Structure

Role: "You are a code auditor. You have a specification and the source files it claims to change."

The prompt contains:
1. Role instruction
2. Spec content (capped 10000 chars)
3. Affected files context (capped per-file 8000, total 50000)
4. Systematic audit checklist (5 steps, see 7.3)
5. Output format (NDJSON findings)

### 7.3 Audit Checklist

```
For EACH affected file, perform these steps in order:

C1. STATE-DEPENDENT BEHAVIORS
Read the file line-by-line. List every state-dependent behavior:
- Conditional disabling: :disabled, disabled, aria-disabled
- Loading states: :loading, loading, spinner, skeleton
- Conditional visibility: v-if, v-show, hidden, display:none, :class with conditions
- Error/success states: error messages, success indicators, color changes
- Computed/derived state: values that depend on other state
For each: what controls it? What are ALL the conditions?

C2. SPEC COVERAGE
For each state-dependent behavior from C1:
- Does the spec mention this element or condition?
- If the spec changes this behavior, does it describe the FINAL state?
- If the spec removes a condition (e.g., removes isPhoneCheckInProgress),
  does it verify no OTHER code still references it?

C3. REMOVAL REACH
For each removal described in the spec (function, variable, condition, import):
- Grep the affected files for ALL references to the removed item
- List every reference found — not just the obvious ones
- Does the spec account for each reference?

C4. ACCEPTANCE COMPLETENESS
For each state-dependent behavior from C1:
- If this behavior is in scope (spec mentions it or should mention it),
  what should the acceptance criteria say?
- Format: "element X shows Y behavior when Z condition, no other condition"
- Check: does any task's acceptance describe this final state?

C5. SIDE EFFECTS
- Could implementing this spec cause any unintended behavior changes?
- Are there shared state dependencies across affected files?
- Are there implicit contracts (function signatures, event names) that could break?
```

### 7.4 Code-Audit Finding Format

NDJSON, one object per line:

```json
{"id":"ca-001","file":"src/components/AddCustomer.vue","line":42,"pattern":":disabled","current_behavior":"isFormLocked || isPhoneCheckInProgress","spec_coverage":"partial","finding":"spec removes isPhoneCheckInProgress but AddCustomer.vue line 42 still references it. phone input has :disabled with both conditions","suggestion":"Add acceptance: phone input :disabled only when isFormLocked, verify isPhoneCheckInProgress removed from all templates","severity":"high","category":"gap"}
{"id":"ca-002","file":"src/components/AddCustomer.vue","line":87,"pattern":":disabled","current_behavior":"isFormLocked || isIdentityCheckInProgress","spec_coverage":"missing","finding":"date-of-birth input :disabled references isIdentityCheckInProgress but spec doesn't mention this binding","suggestion":"Add to scope: remove isIdentityCheckInProgress from date-of-birth :disabled, acceptance should verify single-condition :disabled","severity":"high","category":"gap"}
{"id":"ca-003","file":"src/components/AddCustomer.vue","line":15,"pattern":"v-if","current_behavior":"v-if=\"isPhoneCheckInProgress\"","spec_coverage":"missing","finding":"phone check spinner shown based on isPhoneCheckInProgress but spec doesn't describe what happens to this conditional rendering","suggestion":"Add acceptance: phone check loading indicator removed or replaced","severity":"medium","category":"gap"}
```

### 7.5 Code-Audit Output

```json
{
  "spec": "spec/identity-check-refactor.md",
  "perspective": "code-audit",
  "affected_files": ["src/components/AddCustomer.vue", "src/api/customers.ts"],
  "affected_summary": {
    "total_files": 2,
    "total_lines": 336,
    "chars_included": 12450
  },
  "prompts": [
    {
      "role": "code-auditor",
      "category": "completeness",
      "prompt": "You are a code auditor..."
    }
  ],
  "review_loop": {
    "round": 1,
    "max_rounds": 1,
    "convergence": "single-pass code audit",
    "previous_findings": 0,
    "instruction": "Spawn a sub-agent with this prompt. Collect NDJSON findings. Feed findings back into spec revision or task acceptance updates."
  }
}
```

## 8. Acceptance Quality Lint

### 8.1 New Check: `acceptance-quality`

When running `tp lint <spec.md>`, also check the companion task file for acceptance criteria quality.

Scan all tasks' acceptance strings for weak patterns:

| Pattern | Severity | Message |
|---------|----------|---------|
| Acceptance starts with a past-tense removal verb (removed, deleted, dropped, kaldırıldı, silindi) and contains no present-tense state description | warning | "acceptance describes removal but not final state" |
| Acceptance uses a vague completion verb (corrected, fixed, updated, adjusted, düzenlendi, güncellendi) without specifying what the result is | warning | "acceptance uses vague completion verb without describing result" |
| Acceptance has fewer than 10 words | info | "acceptance is very short, may lack specificity" |

Detection uses case-insensitive matching. The check identifies acceptance strings that describe what was done ("X removed") but don't describe what the result looks like ("input disabled only when form locked"). Only the structure matters — no language-specific word list.

Findings appear under the existing `tp lint` output with rule `acceptance-quality`.

### 8.2 Scope

This check only runs when a companion task file exists (auto-detected from spec path). If no task file, this check is skipped silently.

## 9. Affected Files Scope Lint

### 9.1 New Check: `affected-files-scope`

When running `tp lint <spec.md>`, detect an "Affected Files" table and check for scope descriptions.

Detection: Find a table with a heading containing "affected" and "files" (case-insensitive). Table must have a column containing values matching "modify", "change", "değiştir", or "güncelle" (case-insensitive).

For each row with a "Modify"/"Change" value:
- Check if the row has a cell describing scope (not just the filename)
- A scope cell must contain at least 10 characters of description
- If no scope cell or scope cell is too short → warning

### 9.2 Output

Findings appear under `tp lint` with rule `affected-files-scope`.

## 13. What Doesn't Change

- `tp lint` structural checks unchanged
- `tp validate` line coverage unchanged

- No new dependencies
