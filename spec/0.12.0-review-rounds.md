# Review Multi-Round Support

**Status:** Released
**Date:** 2026-04-02

## Problem

`tp review` generates 3 adversarial prompts per call but is stateless — it has no concept of rounds. Agents doing multi-round review must:

1. Call `tp review spec.md`, get 3 prompts
2. Spawn 3 sub-agents, collect NDJSON findings
3. Fix spec based on findings
4. Manually write "Previous round found and FIXED: ..." context into prompts
5. Call `tp review spec.md` again

Steps 3-4 waste agent tokens and are error-prone. The agent must parse findings, summarize them, and inject the summary into the next round's prompts — all manually.

## Proposed Solution

Add `--round` and `--findings` flags to `tp review` so it generates round-aware prompts automatically:

```bash
tp review spec.md                          # Round 1 (default, unchanged behavior)
tp review spec.md --round 2 --findings r1.ndjson   # Round 2 with previous findings
tp review spec.md --round 3 --findings r1-r2.ndjson # Round 3 with all prior findings
```

tp reads previous findings from NDJSON, deduplicates them, and injects a "previously addressed" block into each prompt. The agent no longer writes round context manually.

## Design Principles

| Principle | Application |
|-----------|-------------|
| P1: AX First | Eliminates manual round-management boilerplate from agent loop |
| P2: Batch Parity | Works the same for round 2 as round 1 — same output shape |
| P3: Minimal Tokens | Findings summary is compact (one line per finding, capped at 50) |
| P4: Agent Plans, Tool Executes | Agent decides when to re-review; tp generates round-aware prompts |

## Command Changes

### New Flags

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--round` | int | 1 | Current review round number (1-indexed) |
| `--findings` | string | "" | Path to NDJSON file with previous round findings |

### Validation Rules

1. `--round` must be >= 1. Error if < 1.
2. `--round 1` with `--findings` is valid (agent may have findings from a prior review session).
3. `--round > 1` without `--findings` produces a warning hint: "round N without --findings: prompts will not exclude previously reported issues" but does not error.
4. `--findings` file must exist. Exit code 2 (usage) if not found.
5. `--findings` file must be valid NDJSON. Each line must parse as `{"severity":"...","category":"...","location":"...","finding":"...","suggestion":"..."}`. Lines that don't parse are skipped with a warning.

### Findings Format

Input NDJSON (same format as review output):

```
{"severity":"high","category":"completeness","location":"## Error Handling","finding":"Missing retry logic","suggestion":"Add retry section"}
{"severity":"medium","category":"ambiguity","location":"line 42","finding":"'appropriate' is vague","suggestion":"Specify exact behavior"}
```

### Deduplication

Findings are deduplicated by `(category, location)` pair. If the same location appears twice with different categories, both are kept.

### Findings Summary Injection

When `--findings` is provided, each of the 3 prompts gets a block prepended before the spec content:

```
IMPORTANT — Previous Review Round(s) Found and Addressed:
The following issues were identified in previous round(s) and have been addressed by the spec author.
DO NOT re-report these issues unless the fix introduced a NEW, DIFFERENT problem.

  [HIGH] completeness — ## Error Handling: Missing retry logic
  [MED] ambiguity — line 42: 'appropriate' is vague
  [LOW] consistency — ## API: Missing versioning

Focus ONLY on issues NOT listed above. If you find no new issues, respond with an empty array (just []).
```

Rules for the summary block:

1. Findings are sorted by severity (critical > high > medium > low), then by category.
2. Severity is abbreviated: `critical` → `[CRIT]`, `high` → `[HIGH]`, `medium` → `[MED]`, `low` → `[LOW]`.
3. Format per line: `[SEV] category — location: finding (truncated to 80 chars)`.
4. Finding text is truncated at 80 characters with `...` if longer.
5. Maximum 50 findings shown. If more, append: `... and N more (omitted for brevity)`.
6. If findings file is empty or contains no valid entries, the block is not injected and no warning is emitted.

### Round 2+ Prompt Modification

In addition to the findings summary block, when `--round >= 2`:

1. Replace the standard opening instruction with a round-aware version:
   - Round 1 default: "You are a senior engineer who must implement this spec tomorrow..."
   - Round 2+: "You are a senior engineer who must implement this spec tomorrow. This is review round {N} — focus ONLY on issues not previously reported."

2. Append to the finding format section: "Remember: only report NEW issues not covered by the previous findings listed above."

### Output Changes

The `review_loop` object in output is updated:

```json
{
  "review_loop": {
    "round": 2,
    "max_rounds": 2,
    "convergence": "no new high-severity findings",
    "previous_findings": 5,
    "instruction": "Spawn sub-agents for each prompt. Collect findings. If any critical/high severity, revise spec and re-run `tp review --round 3 --findings <combined.ndjson>`. Stop after max_rounds or when no new high-severity findings."
  }
}
```

Changes to `review_loop`:

1. New field: `round` (int) — current round number.
2. New field: `previous_findings` (int) — count of loaded previous findings.
3. `instruction` is updated to include `--round N+1 --findings` guidance.

### Behavior Matrix

| `--round` | `--findings` | Behavior |
|-----------|-------------|----------|
| 1 (default) | not provided | Original behavior, no changes |
| 1 | provided | Findings summary injected (pre-loaded from prior session) |
| 2+ | not provided | Warning hint, no findings injection, round-aware prompt opening |
| 2+ | provided | Full behavior: findings summary + round-aware prompts |
| 0 | any | Error: "round must be >= 1" (exit code 2) |

## Edge Cases

1. **Empty findings file**: No injection, no warning. Behaves like `--findings` not provided.
2. **Findings file with only invalid lines**: Warning for each invalid line, then empty findings. No injection.
3. **Findings with missing fields**: Missing fields use empty string. `severity` defaults to `"unknown"`.
4. **Very large findings file**: Read all lines but cap summary at 50 entries. No memory concerns — files are small (NDJSON findings).
5. **Non-existent findings file**: Exit code 2 with clear error: "findings file not found: <path>".
6. **Binary/non-text findings file**: First line fails to parse, warning emitted, treated as empty.
7. **Concurrent access**: `tp review` is read-only (no flock needed). `--findings` file is read, never written.

## Implementation Order

1. Add `--round` and `--findings` flags to `newReviewCmd()`
2. Parse findings file into `[]reviewFinding` struct
3. Deduplicate findings by `(category, location)`
4. Build findings summary string
5. Inject summary block into each prompt
6. Add round-aware prompt opening for `--round >= 2`
7. Update `review_loop` output with `round` and `previous_findings`
8. Add validation (round >= 1, file exists, NDJSON parse)
9. Add tests

## Test Cases

1. **Round 1, no findings**: Identical output to current behavior (backward compat).
2. **Round 2 with findings file**: Prompts contain findings summary block, `review_loop.round` is 2.
3. **Round 2 without findings file**: Output contains warning hint, no findings injection.
4. **Round 0**: Error with exit code 2.
5. **Findings file not found**: Error with exit code 2.
6. **Findings with duplicate (category, location)**: Deduplicated in summary.
7. **Findings with > 50 entries**: Capped at 50 with "and N more" suffix.
8. **Findings with long finding text**: Truncated at 80 chars.
9. **Findings file with invalid lines**: Skipped with warning, valid lines processed.
10. **Empty findings file**: No injection, same as round 1.
