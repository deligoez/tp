# 0.20.0 — Review Enhancements

Review workflow improvements: findings context for regression prevention, orphan list lint rule.
Sections 1 (Review State Directory) and 4 (Diff Auto-Activation) are **deferred** pending agent feedback — documented in CLAUDE.md "Deferred Ideas". Section 3 partially implemented (orphan-list-item done, broken-cross-ref deferred).
## Motivation

In a 21-round review cycle on a 679-line spec:
- Agent managed 42+ NDJSON files manually (findings per round per perspective)
- Round 2+ re-read the entire spec even when only 3 lines changed
- Resolved findings were summarized too briefly — agents re-reported known issues
- No automated "edit hygiene" checks between rounds

## 1. Review Round State Directory

### 1.1 Problem

Agents manually manage findings files: `r1-implementer.ndjson`, `r1-merged.ndjson`, etc.
With 21 rounds × 3 perspectives, this becomes 60+ files to track.

### 1.2 Proposed Solution

tp manages a `.tp-review/` directory alongside the task file:

```
spec/
  feature.md
  feature.tasks.json
  .tp-review/
    feature/
      round-1/
        prompts.json           # Generated prompts
        implementer.ndjson     # Sub-agent findings
        tester.ndjson
        architect.ndjson
        merged.ndjson          # tp review --merge output
      round-2/
        ...
      state.json               # Current round, convergence status
```

### 1.3 Behavior Changes

| Command | Current | Proposed |
|---------|---------|----------|
| `tp review spec.md` | Outputs prompts to stdout | Also saves to `.tp-review/<spec>/round-N/prompts.json` |
| `tp review --round 2` | Requires `--findings` flag | Auto-discovers `round-1/merged.ndjson` if `--findings` omitted |
| `tp review --merge` | Requires output path | Auto-saves to `round-N/merged.ndjson` |
| `tp review --report` | Requires file paths | Auto-discovers all rounds |

### 1.4 state.json Format

```json
{
  "spec": "spec/feature.md",
  "current_round": 3,
  "rounds": [
    {
      "round": 1,
      "started_at": "2026-04-01T10:00:00Z",
      "perspectives": ["implementer", "tester", "architect"],
      "findings_count": 12,
      "resolved_count": 8,
      "high_severity": 3
    }
  ],
  "converged": false
}
```

### 1.5 Open Questions

- Should `.tp-review/` be git-tracked? Likely yes (findings are valuable history).
- How to handle multiple concurrent reviews of the same spec? (edge case)
- Should `tp review --clean` remove the directory?
- Backward compatibility: if no `.tp-review/` exists, fall back to current manual behavior.

## 2. Findings Context Enhancement

### 2.1 Problem

Round N prompt says "69 findings resolved" but the agent doesn't know WHAT was resolved.
It re-reports known issues, wasting rounds.

### 2.2 Current Behavior

`tp review --round N --findings file.ndjson` injects a findings summary into prompts:
- Count of resolved/unresolved by severity
- Brief category breakdown

### 2.3 Proposed Enhancement

Inject the actual findings text for high/critical unresolved items:

```
Previous findings summary:
- 69 findings total: 45 resolved, 24 unresolved
- Unresolved high/critical (DO NOT re-report these):
  #12 [high] Section 3.2: Missing error handling for timeout case
  #15 [critical] Section 4.1: Race condition in concurrent access
  #23 [high] Section 5.3: No validation for negative values
  ...
```

### 2.4 Token Budget

- Cap at 20 findings in the summary (highest severity first)
- Each finding: ~50 tokens (index + severity + one-line summary)
- Total overhead: ~1000 tokens per prompt (acceptable vs re-reporting savings)

### 2.5 Open Questions

- Should resolved findings also be listed (to prevent regression)?
- Should the budget be configurable? (`--findings-budget N`)
- How to handle findings that reference moved/deleted sections?

## 3. Edit Hygiene Lint Rules

### 3.1 Problem

Many review rounds were spent finding "edit hygiene" issues:
- Duplicate consecutive paragraphs (copy-paste artifacts)
- Broken cross-references (`§3.2 step 10` but §3.2 has 9 steps)
- Orphaned numbered list items

These are mechanical checks that don't need a review agent.

### 3.2 Proposed Rules for `tp lint`

| Rule | Severity | Description |
|------|----------|-------------|
| `duplicate-paragraph` | warning | Two consecutive identical paragraphs (not just lines) |
| `broken-cross-ref` | warning | `§X.Y step N` where section X.Y has fewer than N steps |
| `orphan-list-item` | info | Numbered list that starts at >1 or has gaps |

### 3.3 Complexity Assessment

- `duplicate-paragraph`: Medium — need to define paragraph boundaries (blank-line separated)
- `broken-cross-ref`: High — need to parse cross-reference formats (§, "see section", "step N")
- `orphan-list-item`: Low — regex on `^\d+\.` with gap detection

### 3.4 Open Questions

- What cross-reference formats to support? (`§3.2`, `Section 3.2`, `see 3.2`, `step 10 in §3.2`)
- Should broken-cross-ref be opt-in? (High false positive risk)
- Priority ordering: orphan-list-item (easy) → duplicate-paragraph (medium) → broken-cross-ref (future)

## 4. Diff-Based Review Auto-Activation

### 4.1 Problem

`--diff-from` exists but agents don't use it in round 2+.
Every round re-reads the entire spec, wasting ~80-90% of tokens.

### 4.2 Proposed: Automatic Diff Detection

With review state directory (Section 1), tp can:
1. Save a spec snapshot at each round start
2. On round 2+, automatically diff against previous snapshot
3. Only inject changed sections (plus minimal context)

### 4.3 Behavior

```bash
# Round 1: tp saves snapshot to .tp-review/feature/round-1/spec-snapshot.md
tp review spec.md

# Round 2: tp auto-diffs against round-1 snapshot
tp review spec.md --round 2
# Equivalent to: tp review spec.md --round 2 --diff-from .tp-review/feature/round-1/spec-snapshot.md
```

### 4.4 Override

`--spec-inline` forces full spec injection (overrides auto-diff).
`--diff-from <path>` manually specifies the base (overrides auto-snapshot).

### 4.5 Dependency

Requires Section 1 (Review Round State Directory) to be implemented first.
