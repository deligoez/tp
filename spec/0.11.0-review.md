# Spec Review — AI-Powered Specification Quality Check

**Status:** Draft
**Date:** 2026-04-01

## Problem

Agents write specs that look complete but contain gaps: missing edge cases, ambiguous requirements, untested paths, inconsistencies between sections. Users review and push back 5-10 times before the spec is ready for decomposition.

The root cause: the agent that wrote the spec can't effectively review its own output — it has the same blind spots. A fresh perspective (different prompt, different role) catches different issues.

## Proposed Solution

Add `tp review <spec.md>` that generates structured, targeted review prompts based on the spec's content. The agent feeds these prompts to sub-agents (via Claude Code's Agent tool) who review adversarially. Findings are collected and the spec is revised.

tp generates the prompts (it knows the spec structure). The agent executes them (it has AI capabilities). This follows P4: Agent Plans, Tool Executes.

## Design Principles

Based on research into spec review best practices:

| Principle | Source | Application |
|-----------|--------|-------------|
| Checklist-driven > open-ended | IEEE Std 1028 | Each prompt targets one defect category |
| Role-based personas | Prompt engineering research | 3 personas: implementer, tester, architect |
| 2-3 rounds max | AWS evaluator patterns | Hard cap, convergence on no new high-severity |
| Adversarial framing | Constitutional AI | "Find reasons this will fail" not "is this good?" |
| Spec-structure-aware | tp's own lint data | Questions reference specific tables, lists, sections |
| Structured output | Actionability research | severity + category + location + finding + suggestion |

## Prompt Generation

### 3 Review Personas

Each persona gets a targeted prompt with spec-specific questions.

#### 1. Implementer (completeness)

Role: "You are a senior engineer who must implement this spec tomorrow."

Questions generated from spec structure:
- For each **table**: "Table '{heading}' has {rows} rows. For each row, could you implement it without asking clarifying questions? What edge cases are missing?"
- For each **numbered list**: "List '{heading}' has {items} items (#1-#{last}). For each item, is the acceptance criteria testable? What happens when it fails?"
- For each **code block**: "The code snippet at line {line} — does it handle error cases? Are the types/signatures accurate?"
- Generic: "What happens when the happy path fails? Where are the error handling gaps?"

#### 2. Tester (ambiguity + verifiability)

Role: "You are a QA engineer writing tests from this spec."

Questions:
- "For each requirement, could two testers write contradictory tests that both satisfy the spec?"
- For each **table row**: "Can you write a deterministic pass/fail test for this row?"
- For each **numbered list item**: "What's the expected output? What's the boundary?"
- Generic: "Which requirements use vague language ('appropriate', 'reasonable', 'properly')?"

#### 3. Architect (consistency + feasibility)

Role: "You are reviewing this spec for architectural issues before approving implementation."

Questions:
- "Do any sections contradict each other?"
- "Is there a 'What doesn't change' section? If not, what existing behavior could break?"
- "Are there performance implications not addressed?"
- "Does the implementation order match the dependency graph?"
- For specs with multiple lists: "Cross-reference '{list_a}' and '{list_b}' — items in one but not the other?"

### Finding Format

Every sub-agent must respond in this format:

```json
{"severity": "critical|high|medium|low", "category": "completeness|ambiguity|consistency|feasibility|redundancy", "location": "section or line reference", "finding": "what is wrong", "suggestion": "how to fix it"}
```

## Review Loop

### Round 1 (always runs)

1. `tp review spec.md` generates 3 prompts
2. Agent spawns 3 sub-agents (can be parallel)
3. Sub-agents return findings in JSON format
4. Agent collects and deduplicates findings
5. Agent presents findings to user or auto-fixes low/medium severity

### Round 2 (conditional)

Triggered when Round 1 has critical or high severity findings:

1. Spec is revised based on Round 1 findings
2. `tp review spec.md` generates new prompts (spec changed → different questions)
3. Only run prompts for categories that had findings
4. If no new high-severity findings → converged

### Round 3+

Not recommended. If Round 2 still has critical findings, the spec likely needs fundamental rethinking, not incremental fixing. tp should suggest: "2 rounds of review still have critical findings. Consider restructuring the spec."

### Convergence Output

```json
{
  "converged": true,
  "rounds": 2,
  "total_findings": 8,
  "resolved": 6,
  "remaining": [
    {"severity": "low", "finding": "..."}
  ]
}
```

## Integration with Existing tp Workflow

### Updated Workflow A (Decompose)

```
1. Write spec in spec/<version>.md
2. tp lint spec.md → fix structural issues
3. tp review spec.md → review loop (1-2 rounds)
4. Decompose into tasks with source_lines
5. tp import → tp validate → implement
```

Step 3 is new. It sits between lint (structural) and decompose (task creation).

### SKILL.md Update

```
### A: Decompose (spec exists, no .tasks.json)

1. `tp lint <spec.md>` — fix structural issues
2. `tp review <spec.md>` — adversarial review loop:
   - Spawn sub-agents with generated prompts
   - Collect findings, fix spec, re-review if needed
   - Converge within 2 rounds
3. Decompose into tasks with source_lines
4. Backward pass + tp validate
5. tp import
```

## What Doesn't Change

- `tp lint` stays as-is — structural checks only
- `tp validate` stays as-is — task-level validation
- tp does NOT call AI/LLM — it generates prompts, agent executes
- No new dependencies
- No changes to task lifecycle (plan, done, commit, report)
