# AX Improvements — Agent Feedback Quick Wins

**Status:** Draft
**Date:** 2026-04-05

## 1. Problem

Real-world agent usage of tp exposed 10 friction points across all four foundational principles. These are concrete, reproducible issues reported by agents after multi-session implementation cycles. Each one wastes tokens, causes retries, or forces workarounds.

### P3 (Minimal Tokens) Violations

1. **Review prompts embed full spec inline by default.** Each review prompt is ~50KB because spec content is inlined. With 3 personas, a single round costs ~150KB in prompt tokens alone. `--spec-ref` exists but is opt-in. Agents can read files — inline should be opt-in, not the default.

2. **`tp next` output is too verbose.** Returns task + acceptance + spec_excerpt + blocks + quality_gate + remaining. Most agent invocations only need id and acceptance. No `--minimal` flag exists (unlike `tp plan --minimal`).

3. **`tp done` reason length check wastes tokens when `--gate-passed` is set.** The rule `len(reason) >= len(acceptance)/2` forces agents to write long reasons even for simple tasks where `--gate-passed` already attests the quality gate passed. This is pure token waste.

### P2 (Batch Parity) Gaps

4. **`tp done` cannot close multiple tasks in one call without NDJSON.** After parallel work, agents close 5-6 tasks one by one. `--batch` requires writing an NDJSON file first. Multi-ID syntax (`tp done id1 id2 id3 "reason"`) is missing.

### P1 (AX First) Issues

5. **`estimate_minutes` field name is not intuitive.** Agent tried `estimation_minutes` — rejected. The `deps` → `depends_on` alias pattern already exists but isn't applied here.

6. **`acceptance` rejects arrays.** Agents naturally write `["item1", "item2"]` in JSON. tp requires a string. Counter-intuitive for JSON-native consumers.

7. **`--file` flag position error lacks hint.** When an agent writes `tp done task-id "reason" --file path`, Cobra may parse it correctly as a persistent flag, but if it fails, the error doesn't suggest the correct syntax. A hint like "did you mean: tp --file X done ..." would save a retry.

8. **`tp use` command missing for multi-file workflows.** Switching between task files requires `--file` or `TP_FILE` on every call. A `tp use <file>` command that writes a `.tp-active` marker would eliminate this friction.

### Missing Features

9. **`tp audit --affected-files` syntax is unintuitive.** StringArrayVar requires `--affected-files a --affected-files b`. Agents expect `--affected-files a,b,c` (comma-separated). Also, auto-detect via git diff fails silently when there are no unstaged changes but there are committed changes since last tag — the error message doesn't guide toward `--base`.

10. **No auto-verify hint after `tp review --resolve-all`.** After resolving all findings, agents must remember to run `tp review --verify` manually. A hint or `--auto-verify` flag would close this gap.

## 2. Root Cause

tp was designed with excellent core architecture (2-call pattern, structured commits, spec lifecycle) but several API surfaces were designed from the implementer's perspective rather than the agent consumer's. Field naming, flag defaults, and output verbosity were not optimized through the AX lens of P3 (Minimal Tokens).

## 3. Proposed Solution

Ten changes organized by implementation complexity:

| # | Change | Affected Files | Complexity |
|---|--------|---------------|------------|
| A | Flip `--spec-ref` to default, add `--spec-inline` | `review.go` | Trivial |
| B | Add `--minimal` flag to `tp next` | `next.go` | Easy |
| C | Skip reason length check when `--gate-passed` | `closure.go` | Trivial |
| D | Multi-ID support for `tp done` | `done.go` | Medium |
| E | `estimation_minutes` alias in UnmarshalJSON | `task.go` | Trivial |
| F | Accept `acceptance` as string or `[]string` | `task.go` | Easy |
| G | Better error hint for `--file` flag position | `discover.go` | Easy |
| H | `tp use <file>` command with `.tp-active` marker | `cli/use.go`, `discover.go` | Medium |
| I | Comma-separated `--affected-files` + better error messages | `audit.go` | Easy |
| J | Auto-verify hint after `--resolve-all` | `review_resolve.go` | Trivial |

## 4. Design Principles

| # | Principle | Application |
|---|-----------|-------------|
| P1 | AX First | Every change removes a documented agent friction point |
| P2 | Batch Parity | Multi-ID done (#D) makes N closures as easy as 1 |
| P3 | Minimal Tokens | Spec-ref default (#A), next --minimal (#B), reason relaxation (#C) cut output |
| P4 | Agent Plans, Tool Executes | All changes are deterministic — no new orchestration |

### What Doesn't Change

- Task file format is unchanged
- `--compact` behavior is unchanged
- 3-state workflow (open → wip → done) is unchanged
- Single-ID `tp done` output shape is unchanged (`closed` remains a string)

### Breaking Changes

1. **`--spec-ref` removed, `--spec-inline` added.** `tp review` now omits inline spec by default (reference mode). The old `--spec-ref` flag is removed entirely — not deprecated, not kept as a no-op. The new `--spec-inline` flag opts into inline embedding. Any automation relying on inline spec content must add `--spec-inline`.

2. **Multi-ID `tp done` returns different output shape.** When 2+ task IDs are passed, `closed` is an array and `failed` array is added. Single-ID calls are unchanged (string). Consumers can detect multi-ID by checking if `closed` is a string or array.

## 5. Detailed Design

### 5.A — Flip `--spec-ref` Default

**Current:** `--spec-ref` is `false` (inline is default).
**New:** `--spec-ref` is `true` (reference is default), `--spec-inline` opts into inline.

#### Changes

1. In `review.go`, remove `--spec-ref` flag definition (line 169). Add `--spec-inline` flag (default: `false`).
2. Invert all conditionals: `if specRef` → `if !specInline`, `if !specRef` → `if specInline`.
3. Update `validateModeFlags` to replace `specRef` checks with `specInline` checks.
4. Update `runReviewVerify` in `review_verify.go` — change parameter from `specRef bool` to `specInline bool`, invert logic.
5. Update all test files referencing `--spec-ref` to use `--spec-inline`.
6. Update error message string in `validateModeFlags` from `"--spec-ref"` to `"--spec-inline"`.
7. Update help text and examples to reflect the new default.

#### Acceptance Criteria

- `tp review spec.md` outputs prompt with spec path reference (no inline content)
- `tp review spec.md --spec-inline` outputs prompt with full inline spec content
- `--spec-ref` flag is removed (clean break, not deprecated) — using it produces Cobra "unknown flag" error (exit 2)
- `--verify` mode works correctly with new default (spec-ref behavior)
- `validateModeFlags` rejects `--spec-inline` with incompatible modes (same rules as old `--spec-ref`)

### 5.B — `tp next --minimal`

**Current:** `tp next` returns `{task, blocks, remaining, quality_gate, spec_excerpt}` where `task` is the full task object.
**New:** `tp next --minimal` returns `{id, acceptance}` only.

#### Changes

1. Add `--minimal` flag to `next` command (same pattern as `plan --minimal`).
2. When `--minimal` is set, output only `{"id": "...", "acceptance": "..."}` — no task envelope, no blocks, no remaining, no quality_gate, no spec_excerpt.
3. `--minimal` always outputs JSON regardless of TTY detection (agent-facing flag).

#### Acceptance Criteria

- `tp next --minimal` returns `{"id": "...", "acceptance": "..."}`
- `tp next` (without --minimal) returns full output unchanged
- `tp next --minimal --peek` works (preview + minimal)
- Claiming behavior is unchanged (--minimal only affects output shape)
- `--minimal` forces JSON output even in TTY mode
- Acceptance is returned as the stored string, regardless of how it was originally input (e.g., array → joined string)
- `tp next --minimal` when no claimable task exists returns the same error/output as `tp next` (--minimal only affects success output shape)

### 5.C — Relax Reason Length When `--gate-passed`

**Current:** `VerifyClosure` requires `len(reason) >= len(acceptance)/2` even when `gatePassed` is true.
**New:** When `gatePassed` is true, skip the minimum length check entirely.

#### Changes

1. In `closure.go`, move the `if gatePassed { return nil }` early-return (currently line 98) to **after** the forbidden pattern check (line 83) but **before** the minimum length check (line 91). This preserves forbidden pattern enforcement while skipping the length check.

#### Acceptance Criteria

- `tp done id "tests pass" --gate-passed` succeeds even if reason is short
- `tp done id "tests pass"` (without --gate-passed) still enforces minimum length
- `tp done id "deferred" --gate-passed` still fails (forbidden pattern check preserved)
- `--covered-by` behavior unchanged

### 5.D — Multi-ID `tp done`

**Current:** `tp done <id> [reason]` — single ID only.
**New:** `tp done <id1> [id2...] <reason>` — last positional argument is **always** the reason, all preceding are task IDs. Shared reason and flags apply to all.

#### Argument Parsing Algorithm

1. Collect all non-flag positional args in order.
2. **Last arg is always the reason** — no task-ID sniffing, no file lookup needed for parsing.
3. All preceding args are task IDs.
4. If only 1 arg and no `--stdin`/`--reason-file`: this is the task ID (reason resolved from other source or error).
5. If 0 args: error (task ID required).
6. When `--stdin` or `--reason-file` is provided: **all** positional args are task IDs (no positional reason).
7. Standard Cobra flag parsing applies — flags can appear anywhere among positional args.

#### Syntax

```bash
tp done task1 "evidence"                    # single (unchanged)
tp done task1 task2 task3 "shared evidence"  # multi-ID: last arg = reason
tp done task1 task2 --gate-passed "evidence" # multi-ID with flags
tp done task1 task2 task3 --reason-file e.txt # all args are IDs (reason from file)
```

#### Output Shape

**Single-ID** (1 task ID): output is **unchanged** — `{"closed": "task-id", "remaining": {...}, "has_next": bool}`. This preserves full backward compatibility.

**Multi-ID** (2+ task IDs): output uses arrays — `{"closed": ["id1","id2"], "failed": [{"id":"id3","error":"...","hint":"..."}], "remaining": {...}, "has_next": bool}`.

Consumers can detect the mode by checking if `closed` is a string (single) or array (multi).

#### Processing Semantics

- Tasks are processed **in argument order** within a single file lock.
- Each closure is visible to subsequent closures in the same batch (closing task1 can unblock task2 if task2 depends on task1).
- If a task fails (state error, validation error), it goes into the `failed` array. Previously closed tasks in the same batch remain closed.
- `--auto-commit` is **not supported** with multi-ID. If both multi-ID and `--auto-commit` are provided, exit with usage error: `"--auto-commit is not supported with multiple task IDs. Use tp done --batch for multi-task auto-commit."` Rationale: commit message format and failure rollback semantics are too complex for the multi-ID shorthand. `--batch` already supports this.

#### Exit Codes

- Exit 0: all tasks closed successfully, or partial failure (at least one closed, at least one failed). Agents must check the `failed` array when exit 0.
- Exit 1: all tasks failed (any failure type — validation, state, or mixed). Simplest rule: if zero tasks closed, exit 1.
- Exit 2: usage error (`--auto-commit` with multi-ID, `--batch` with positional args, missing reason).

#### Changes

1. Change `cobra.RangeArgs(0, 2)` to `cobra.ArbitraryArgs`. Validate arg count manually inside `runDone` **after** the `--batch` check (which uses 0 positional args). This preserves `tp done --batch file.ndjson` working with 0 args.
2. Reject `--batch` when positional args are present: `"--batch is mutually exclusive with positional task IDs"` (exit 2).
3. Refactor `resolveReason` to accept the full args slice and return `(taskIDs []string, reason string, err error)`. When `--stdin`/`--reason-file` is set, all args are IDs. Otherwise, last arg is reason, rest are IDs. If only 1 arg and no `--stdin`/`--reason-file`: error `"reason is required (use positional reason, --stdin, or --reason-file)"`.
4. Reject `--auto-commit` when len(taskIDs) > 1 with usage error: `"--auto-commit is not supported with multiple task IDs. Use tp done --batch for multi-task auto-commit."` (exit 2).
5. For single-ID: existing code path, existing output shape (string `closed`).
6. For multi-ID: loop over taskIDs in order, collect results, output arrays.
7. Use `make([]string, 0)` for closed and `make([]map[string]any, 0)` for failed to avoid nil in JSON.

#### Acceptance Criteria

- `tp done id1 id2 "reason" --gate-passed` closes both tasks
- `tp done id1 "reason"` returns `{"closed": "id1", ...}` (string, backward-compatible)
- `tp done id1 id2 "reason"` returns `{"closed": ["id1","id2"], "failed": [], ...}` (arrays)
- Partial failure returns both closed and failed arrays with exit 0
- `tp done id1 id2 --auto-commit "reason"` exits with usage error (exit 2)
- `tp done id1 id2 --reason-file r.txt` reads reason from file, both args are IDs
- `tp done id1 id2 id3 --stdin` reads reason from stdin, all 3 args are IDs
- `--covered-by` with multi-ID applies same reference to all
- When closing task2 depends on task1 and both are passed in order, task1 closes first enabling task2
- `tp done --batch file.ndjson task1 "reason"` exits with usage error (exit 2) — `--batch` mutually exclusive with positional args
- `tp done task1 task1 "reason"` — first closes, second fails as "already done" (partial success, exit 0)
- `--covered-by` referencing a task in the same batch is valid only if the referenced task appears earlier and closes successfully
- `--gate-passed` sets `gate_passed_at` on each successfully closed task. Failed tasks are not modified.
- `tp done --reason-file r.txt` (no task ID) exits with usage error (exit 2)
- Use case guidance: multi-ID for 2-5 tasks with shared evidence; `--batch` for larger sets or per-task evidence

### 5.E — `estimation_minutes` Alias

**Current:** Only `estimate_minutes` accepted.
**New:** `estimation_minutes` also accepted (same pattern as `deps` → `depends_on`).

#### Changes

1. In `task.go` `UnmarshalJSON`, add `EstimationMinutes int` to aux struct with JSON tag `json:"estimation_minutes"`.
2. If `EstimateMinutes` is 0 and `EstimationMinutes` is non-zero, copy the value. Note: `estimate_minutes: 0` is not a valid value (already rejected by `validate` rule "estimate_minutes is required"), so the zero-value check is safe.

#### Acceptance Criteria

- `tp add '{"id":"x","estimation_minutes":5,...}'` succeeds
- `tp import` with `estimation_minutes` in task JSON succeeds
- If both `estimate_minutes` and `estimation_minutes` are present, `estimate_minutes` wins
- Existing `estimate_minutes` usage is unchanged

### 5.F — Accept `acceptance` as Array

**Current:** `acceptance` must be a JSON string.
**New:** Accept both string and `[]string`. Array is joined with `\n- ` prefix.

#### Changes

1. In `task.go` `UnmarshalJSON`, add `AcceptanceRaw json.RawMessage` to aux struct (exclude `Acceptance` from Alias unmarshal).
2. After unmarshal, try `json.Unmarshal(raw, &stringVal)` first. If that fails, try `json.Unmarshal(raw, &arrayVal)` with type `[]string`.
3. If array: reject if empty (acceptance required). Reject if any element is empty string. Join with `"- " + strings.Join(arr, "\n- ")`.
4. If neither string nor `[]string` parses: return error `"acceptance must be a string or array of strings"`.
5. Validation continues to check non-empty string after normalization.

#### Type Restrictions

- Only `string` and `[]string` are accepted. No nested arrays, no mixed types, no numbers, no nulls.
- Whitespace-only strings in arrays are treated as empty (rejected).
- Single-element array `["one"]` produces `"- one"`.
- Array form is **input-only**. Internal storage and all output serialization uses the joined string form. Round-tripping through `tp show` returns the string form, not the original array.
- **ParseAcceptanceCriteria alignment:** Update `ParseAcceptanceCriteria` in `closure.go` to also split on `"\n- "` (bullet list delimiter) in addition to existing `". "` and `"; "` delimiters. This ensures array-input acceptance is split into individual criteria for closure verification.

#### Acceptance Criteria

- `tp add '{"acceptance":"single string",...}'` unchanged
- `tp add '{"acceptance":["item1","item2"],...}'` produces `"- item1\n- item2"`
- `tp add '{"acceptance":["one"],...}'` produces `"- one"`
- Empty array `[]` is rejected (acceptance required)
- Array with empty string `["a","","b"]` is rejected
- Nested array `[["a"]]` is rejected with clear error
- `tp import` with array acceptance works
- Internal storage is always string (no schema change)

### 5.G — Better `--file` Flag Hint

**Current:** When `--file` is misplaced or task file discovery fails, error message is generic.
**New:** Enhance discovery error to suggest `--file` and `TP_FILE` usage.

#### Changes

1. In `discover.go`, when multiple files found, include: `"hint: set TP_FILE=<path> or use tp --file <path> <command>"`.
2. In `discover.go`, when no files found, include: `"hint: run tp init <spec.md>, or set TP_FILE=<path>"`.
3. These hints are already partially present — refine wording for clarity.

#### Acceptance Criteria

- "multiple task files found" error includes `TP_FILE` and `--file` hint
- "no task file found" error includes `tp init` and `TP_FILE` hint
- Error messages are concise (under 200 chars each)

### 5.H — `tp use <file>` Command

**Current:** Task file selection requires `--file` flag or `TP_FILE` env var on every call.
**New:** `tp use <file>` writes a `.tp-active` file in CWD. Discovery checks `.tp-active` before directory scan.

#### Discovery Priority (Updated)

1. `--file` flag (explicit per-call)
2. `TP_FILE` env var (session-level)
3. `.tp-active` file in CWD (project-level, persisted)
4. Directory scan for `*.tasks.json`

#### Commands

```bash
tp use spec-ax.tasks.json    # set active task file
tp use --clear               # remove .tp-active (idempotent — succeeds even if no .tp-active)
tp use                       # show current active file (or "none")
```

#### Output Format

- `tp use <file>`: on success, prints path to stderr (TTY) or outputs `{"active_file": "path"}` (JSON when piped). Exit 0.
- `tp use --clear`: silent success. Exit 0.
- `tp use` (no args): prints path or "no active task file" to stdout (TTY), or outputs `{"active_file": "path"}` / `{"active_file": null}` (JSON when piped).
- `tp use --clear <file>`: usage error (exit 2) — mutually exclusive.

#### Path Handling

- `tp use` stores the path **as given** by the user (single line, no JSON).
- At discovery time, `.tp-active` is read from **CWD only** (no parent directory traversal).
- Relative paths in `.tp-active` are resolved relative to the directory containing `.tp-active` (which is CWD).
- If the resolved path does not exist at discovery time, error with: `"task file from .tp-active not found: <path>. Run tp use --clear or tp use <new-file>"`.
- `.tp-active` should be added to `.gitignore` — it is user/session-specific like `.env`. The implementation does NOT auto-modify `.gitignore` but the documentation should recommend it.

#### Changes

1. New file: `cli/use.go` — `tp use` command.
2. `discover.go` — check `.tp-active` after env var, before dir scan. Read file, resolve path, validate existence.
3. Exit code 3 (file error) when file specified in `tp use` does not exist.
4. `tp use` only checks file existence — does not validate content (no JSON parsing).
5. No file locking for `.tp-active` writes — it is a project-level config rarely changed concurrently.

#### Acceptance Criteria

- `tp use foo.tasks.json` creates `.tp-active` with path "foo.tasks.json"
- `tp use ../other/bar.tasks.json` creates `.tp-active` with path "../other/bar.tasks.json"
- Subsequent `tp plan` resolves the path relative to `.tp-active` location and uses it
- `tp use --clear` removes `.tp-active` (idempotent — succeeds even if no `.tp-active`)
- `tp use --clear foo.tasks.json` exits with usage error (exit 2)
- `tp use` (no args, no --clear) prints current active file or "no active task file"
- `--file` flag still overrides `.tp-active`
- `TP_FILE` still overrides `.tp-active`
- Non-existent file in `tp use` exits with code 3
- `.tp-active` pointing to later-deleted file produces clear error with hint at discovery time
- `.tp-active` is only checked in CWD, not parent directories

### 5.I — Improved `--affected-files` for Audit

**Current:** `--affected-files` is `StringArrayVar` — requires `--affected-files a --affected-files b`.
**New:** Additionally accept comma-separated values. Improve error messages for auto-detect failures.

#### Changes

1. In `audit.go`, after parsing `--affected-files`, split each value on commas, trim whitespace from each element, filter empty elements, and flatten. Note: no escape mechanism for literal commas in file paths — use repeated `--affected-files` flag form for paths containing commas.
2. Improve error messages:
   - When no changes detected without `--base`: add hint about `--base <tag>` for committed changes.
   - When only binary/markdown files found: list the skipped file types.

#### Acceptance Criteria

- `--affected-files a.go,b.go` works (comma-separated)
- `--affected-files a.go --affected-files b.go` still works (repeated flag)
- `--affected-files a.go,b.go --affected-files c.go` works (mixed)
- Error when no changes detected includes `--base` hint
- Error when only binary/markdown includes list of skipped file extensions
- Auto-detect behavior unchanged when `--affected-files` is not provided

### 5.J — Auto-Verify Hint After `--resolve-all`

**Current:** `--resolve-all` outputs counts and exits. Agent must remember to verify.
**New:** Output includes a `next_step` hint suggesting `tp review --verify`.

#### Changes

1. In `review_resolve.go`, after successful `--resolve-all`, add `next_step` field to JSON output.
2. The `next_step` value is a template string: `"tp review --verify <spec> --findings <file>"` where `<spec>` is a placeholder (literal `<spec>`) and `<file>` is the actual findings file path. The agent fills in the spec path. Rationale: findings NDJSON has no metadata about which spec produced them, and adding a metadata convention is out of scope for this change.
3. For `--resolve` (single finding), after resolving, re-scan all findings in the file. If none remain unresolved, include `next_step` in output. If unresolved findings remain, omit `next_step`.

#### Output Shape

```json
{
  "resolved_count": 5,
  "skipped_count": 0,
  "status": "fixed",
  "file": "findings.ndjson",
  "next_step": "tp review --verify <spec> --findings findings.ndjson"
}
```

#### Acceptance Criteria

- `--resolve-all` output always includes `next_step` with findings file path and `<spec>` placeholder (even when all findings were already resolved — skipped_count > 0, resolved_count = 0)
- `--resolve` (single) includes `next_step` only when all findings are now resolved
- `--resolve` (single) omits `next_step` when unresolved findings remain
- JSON output shape is backward-compatible (new field only, never null — omitted when absent)

### Output Shape Compatibility (5.D)

Single-ID `tp done` output is **unchanged**: `"closed": "task-id"` (string). Multi-ID output uses arrays: `"closed": ["id1","id2"]`. Consumers can detect the mode by checking if `closed` is a string or array. No version field is needed — the output shape directly corresponds to the input mode (1 ID vs 2+ IDs).

## 8. Non-Goals

- **Review loop automation** (`--auto-loop`): Too complex for this release. Requires agent spawning which violates P4. Tracked separately.
- **Title conjunction enforcement changes**: Already a warning by default, error only with `--strict`. No change needed.
- **`tp report` improvements**: Not in feedback scope.
- **New task states**: 3-state workflow is sufficient.
