# 0.21.0 — Skill Interview Phase & SKILL.md Optimization

## Summary

Add a pre-spec interview phase to the tp skill workflow so the agent resolves all ambiguities before writing the spec. Restructure SKILL.md for token efficiency by splitting into SKILL.md (90-120 lines) + REFERENCE.md. Introduce convergence parameters so the user controls quality gates upfront and agents cannot short-circuit review/audit loops.

## Motivation

### Problem 1: Agents Skip Straight to Spec Writing

The current SKILL.md Workflow A starts with `tp lint`. There is no guidance for the agent to first understand the problem space, ask clarifying questions, or resolve design decisions. Agents jump into spec writing with incomplete understanding, leading to:
- Specs with ambiguities that surface during review (wasted review rounds)
- Design decisions made implicitly by the agent without user input
- Rework when the user's intent differs from the agent's assumptions

### Problem 2: Agents Short-Circuit Review/Audit Loops

The current SKILL.md says "Stop when no new high/critical severity findings. Typically 2-3 rounds." Agents interpret this loosely — sometimes stopping after a single clean round, or skipping review entirely when they feel confident. There is no mechanism for:
- The user to specify their quality standard upfront
- Enforcing consecutive clean rounds (one clean round may be a fluke)
- Consistent convergence criteria across review and audit

### Problem 3: SKILL.md Token Overhead

The current SKILL.md is 376 lines (~17KB). Every skill invocation loads the entire file into the agent's context. The command reference tables (lines 232-314) are rarely needed during execution — the agent already knows tp commands from prior use. This wastes ~270 lines of context per invocation.

### Problem 4: tp's Core Philosophy Is Under-Articulated

tp's interview phase embodies a key principle: **resolve all human-dependent decisions before implementation so the agent never needs to stop during execution.** This is the inverse of HITL — tp eliminates the need for human intervention during implementation by front-loading all decisions.

## Design

### 1. Interview Phase (SKILL.md Workflow Change)

Add "Step 0: Interview" to Workflow A, before spec writing.

#### 1.1 Interview Process

The agent MUST complete an interview phase before writing or editing a spec. The process:

1. **Locate starting material.** A "draft spec" is a `.md` file explicitly provided by the user, or a file passed as an argument to the skill invocation. If no file is provided, ask the user to describe the problem and desired solution.
2. **Explore the codebase.** Read the project's CLAUDE.md/README and the files most likely affected by the spec (as identified from the spec text or user description). Limit exploration to files directly referenced or plausibly touched — do not explore the entire codebase.
3. **Identify ambiguities.** List all unclear, ambiguous, or under-specified aspects found in the draft spec or user description.
4. **Ask questions one at a time.** For each ambiguity, ask the user a single focused question. Wait for the answer before asking the next question. Derive follow-up questions from each answer, repeating until no new questions arise.
5. **Prefer codebase over user.** If a question can be answered by reading the codebase (e.g., current behavior, existing patterns), explore instead of asking. Limit exploration to 3-5 distinct files per question (re-reading a previously read file does not count) — if the answer is not found, ask the user. Questions requiring architectural intent or product decisions always go to the user.
6. **Recommend answers.** For each question, provide a recommended answer based on codebase context. If the codebase provides no relevant context, state that and still provide a best-guess recommendation.
7. **Handle non-answers.** If the user says "doesn't matter", "skip", "whatever", provides an empty response, or explicitly defers the decision, treat the agent's recommended answer (from step 6) as accepted and proceed to the next question.
8. **Termination.** The interview is complete when: (a) every claim about behavior in the draft spec (if provided) or in the user's verbal description has been verified against the codebase or confirmed by the user, (b) every design choice with user-visible impact (CLI output, file format, command behavior) has been explicitly decided, and (c) no new questions arise from any previous answer.

#### 1.2 Convergence Parameter Collection

When the agent has no more questions, it MUST collect convergence parameters before proceeding:

1. Ask: "How many consecutive clean review rounds before proceeding to decomposition? (default: 2)" — accept only positive integers in range 1-10. If the user provides a value outside 1-10, explain the valid range and ask again. If the user provides an invalid value twice consecutively, use the default (2) and announce: "Invalid input — using default of 2." If the user declines or skips, use default 2.
2. Ask: "How many consecutive clean audit rounds after implementation? (default: 2)" — same rules: positive integer 1-10, re-ask once if out of range, fallback to default 2 on second invalid or skip.

The agent holds these values in memory until `tp init` is called (see §1.5).

#### 1.3 Interview Exit Announcement

After convergence parameters are collected, the agent announces:

> "I have no more questions. I will now write the spec and review it until **N** consecutive clean review rounds. After implementation, I will audit until **M** consecutive clean audit rounds."

Where N = the agreed `review_clean_rounds` and M = the agreed `audit_clean_rounds`.

#### 1.4 Interview Re-Entry

If the agent discovers new ambiguities while writing the spec, it MUST pause spec writing and return to step 3 (Identify ambiguities) to formally list the new ambiguity, then proceed through steps 4-8. Do NOT re-ask convergence parameters — use the previously agreed values.

#### 1.5 Convergence Parameter Persistence

Convergence parameters are collected during the interview (§1.2) but the task file does not yet exist — it is created later by `tp init` after the spec is written and reviewed.

The agent MUST:
1. Hold the agreed values in memory during spec writing and review.
2. After `tp init <spec.md>` creates the task file, update the workflow section if the user chose non-default values: `tp set --workflow review_clean_rounds=N audit_clean_rounds=M` (see §2.4).
3. If the user accepted defaults (2, 2), skip step 2 — `tp init` already writes both fields with default value 2.

### 2. Convergence Parameters (CLI Change)

#### 2.1 New Workflow Fields

Add two fields to the `workflow` object in `TaskFile`:

```json
{
  "workflow": {
    "quality_gate": "go test ./... && golangci-lint run",
    "review_clean_rounds": 2,
    "audit_clean_rounds": 2
  }
}
```

| Field | Type | Default | Range | Description |
|-------|------|---------|-------|-------------|
| `review_clean_rounds` | integer | 2 | 1-10 | Consecutive review rounds with zero findings (any severity) required before proceeding to decomposition |
| `audit_clean_rounds` | integer | 2 | 1-10 | Consecutive audit rounds with zero findings (any severity) required before declaring implementation complete |

"Finding-free" means **zero findings of any severity** — not just zero critical/high. A round is clean when the reviewing sub-agent produces zero findings (empty NDJSON output or explicit NO_FINDINGS response).

#### 2.2 Validation Rules

- Both fields MUST be positive integers in range 1-10.
- `tp validate` rejects values outside this range with a parameterized error: `"<field> must be between 1 and 10"` (e.g., `"review_clean_rounds must be between 1 and 10"`, `"audit_clean_rounds must be between 1 and 10"`).
- `tp validate` includes convergence values in its output under the existing `workflow` key.

Success output:
```json
{
  "valid": true,
  "workflow": {
    "quality_gate": "go test ./... && golangci-lint run",
    "review_clean_rounds": 2,
    "audit_clean_rounds": 2
  }
}
```

Failure output (convergence field out of range):
```json
{
  "valid": false,
  "errors": [
    "review_clean_rounds must be between 1 and 10"
  ],
  "workflow": {
    "quality_gate": "go test ./... && golangci-lint run",
    "review_clean_rounds": 99,
    "audit_clean_rounds": 2
  }
}
```

The `workflow` key is always included in `tp validate` output regardless of validity — it shows the post-deserialization values (after defaults are applied).

- When fields are **absent** from JSON (pre-migration files), defaults are applied at **struct deserialization time** (Go `UnmarshalJSON`): missing field → default 2. This is a migration convenience.
- When fields are **explicitly set to 0** in JSON, this is an invalid value (0 is outside range 1-10). `UnmarshalJSON` uses `*int` (pointer) internally to distinguish absent (nil → default 2) from explicit-zero (non-nil, value 0 → preserved as 0). Validation in `engine/validate.go` then rejects explicit 0. This is a **validate-time error only** — commands like `tp plan`, `tp list`, `tp show` that load the file will NOT error on explicit 0; only `tp validate` reports the range violation. This avoids breaking all commands when a file contains a bad value.
- `tp validate --strict` does NOT flag missing (absent) convergence fields — defaults are silently applied.
- `tp init` always writes both fields explicitly with default value 2.

#### 2.3 Display in `tp plan`

`tp plan` already passes through the entire `workflow` object from the task file. Since the new fields are part of the `Workflow` struct and populated at deserialization, they appear automatically in plan output. **No code change needed in cli/plan.go** — this is a verification item, not an implementation item.

Behavior under output flags:
- `--json` (default when piped): `workflow` block includes both fields.
- `--compact`: `workflow` block is included (it is operational, not metadata).
- `--minimal`: `workflow` block is **omitted**. This is the existing behavior — `tp plan --minimal` and `tp next --minimal` output only `{id, acceptance}` per task and do not include the workflow block. No code change needed.

#### 2.4 `tp set --workflow` (New Subcommand Form)

`tp set` currently operates on task-level fields (`tp set <id> field=value`). Workflow fields are file-level, not task-level. Add a new form:

```bash
tp set --workflow review_clean_rounds=3 audit_clean_rounds=2
```

- `--workflow` flag switches `tp set` to operate on the `TaskFile.Workflow` object instead of a task.
- Accepts one or more `field=value` pairs. If no pairs are provided, exit with usage error: `"tp set --workflow requires at least one field=value pair"`.
- Validates the same rules as §2.2 (range 1-10 for convergence fields).
- Rejects truly unknown fields with: `"unknown workflow field: <field>"`.
- Rejects known-but-uneditable fields with a distinct error: `"<field> is read-only; edit the task file directly to change it"`. Currently `quality_gate` is the only read-only workflow field.
- Editable workflow fields via `--workflow`: `review_clean_rounds`, `audit_clean_rounds` only. The `quality_gate` field is not editable via `tp set --workflow` in this spec — editing `quality_gate` requires direct JSON editing. This avoids scope creep; `quality_gate` editability can be added in a future spec if needed.

Success output (echoes only the fields that were supplied, not the full workflow):
```json
{"updated": {"review_clean_rounds": 3}}
```

Failure output:
```json
{"error": "review_clean_rounds must be between 1 and 10"}
```

### 3. SKILL.md Split (Skill File Change)

#### 3.1 New Structure

```
skills/tp/
  SKILL.md         90-120 lines — workflows, interview, convergence, closure rules
  REFERENCE.md     remainder — command tables, field aliases, NDJSON format, examples
```

The 90-120 line target is a **guideline**, not a hard constraint. The goal is to keep SKILL.md under 120 lines; exceeding it slightly for clarity is acceptable.

#### 3.2 SKILL.md Content

The restructured SKILL.md is a **complete rewrite**, not a reorganization of the current file. All existing Workflow A/B/C prose, decomposition rules, and closure rules are rewritten from scratch to fit the line budgets below. The current SKILL.md content serves as source material, not as a template to preserve. Approximate line budgets:

1. **Frontmatter + Activation** (~5 lines) — name, description (updated), trigger conditions
2. **Workflow A: Decompose** (~25 lines) — interview → spec → review (with convergence) → decompose → import → validate. All 8 interview steps from §1.1 MUST be represented (combined where natural, but every step must be traceable). The termination criterion from step 8 MUST be preserved verbatim or equivalent. Convergence enforcement is applied by forward-reference to §3.2 item 7.
3. **Workflow B: Execute** (~15 lines) — plan → implement → commit → done → audit (with convergence). Audit operational steps (how to run `tp audit`, spawn sub-agents) remain inline — they are required during execution. Convergence enforcement forward-references item 7.
4. **Workflow C: Resume** (~5 lines) — same as B, plan excludes done tasks. MUST explicitly reference the Convergence Enforcement section (item 7) and state that audit convergence applies equally when resuming.
5. **Decomposition Rules** (~15 lines) — atomicity, concern axes, source_lines, backward pass
6. **Closure Rules** (~10 lines) — evidence requirements, gate_passed, covered_by
7. **Convergence Enforcement** (~10 lines) — the NON-NEGOTIABLE blocks from §4 appear here verbatim as a standalone section after Closure Rules. Workflows A, B, and C forward-reference this section (e.g., "see Convergence Enforcement below").
8. **Reference pointer** (~2 lines) — "For command details, field aliases, NDJSON format, and batch operations: see [REFERENCE.md](REFERENCE.md)"

Moved to REFERENCE.md (structured as sections in this order):

1. **Key Commands** — command reference tables (Primary, Incremental, Task State, Query, Spec & Validation, Data) — currently SKILL.md lines 232-314
2. **Global Flags** — flag table — currently lines 316-323
3. **Acceptance Criteria Format** — delimiter details — currently lines 152-168
4. **JSON Field Aliases** — deps, estimation_minutes, acceptance array — currently lines 170-174
5. **NDJSON Result Format** — batch result format — currently lines 176-187
6. **Task File Discovery** — discovery priority, tp use — currently lines 202-214
7. **Phase Management** — tag-based phase organization — currently lines 345-360
8. **Batch Close Details** — dependency ordering, toposort — currently lines 361-376
9. **Review File Management** — findings file conventions — currently lines 73-83

Note: Audit workflow operational steps (how to run `tp audit`, spawn sub-agents) stay in SKILL.md Workflow B because they are needed during execution. Only detailed format docs and examples move to REFERENCE.md.

#### 3.3 Updated Description

Current:
```
Spec-to-task lifecycle manager for AI coding agents. Decomposes specs into atomic tasks, manages execution order via dependency graph, and batch-closes with evidence.
```

Proposed:
```
Spec-to-task lifecycle manager for AI coding agents. Interviews user to resolve ambiguities, decomposes specs into atomic tasks, manages execution order via dependency graph, and batch-closes with evidence. Use when user wants to implement a spec, plan tasks, decompose a feature, or a *.tasks.json file exists.
```

The `*.tasks.json` trigger activates Workflow B (Execute) directly, bypassing the interview phase. Interview is only part of Workflow A (Decompose).

### 4. Convergence Enforcement Language (SKILL.md)

The following enforcement rules MUST appear verbatim in SKILL.md §3.2 item 7 (Convergence Enforcement section):

#### 4.1 Review Convergence

> **NON-NEGOTIABLE:** You MUST NOT proceed to decomposition until you have completed N consecutive review rounds with zero findings (any severity), where N = `workflow.review_clean_rounds` (default: 2). A single clean round is insufficient — consecutive clean rounds confirm the spec is stable. Do not skip rounds, summarize findings as "minor", or declare convergence prematurely.

#### 4.2 Audit Convergence

> **NON-NEGOTIABLE:** You MUST NOT declare implementation complete until you have completed N consecutive audit rounds with zero findings (any severity), where N = `workflow.audit_clean_rounds` (default: 2). This applies equally when resuming via Workflow C. Do not skip rounds or declare the audit passed based on your confidence in the implementation.

#### 4.3 Interview Completeness

> **NON-NEGOTIABLE:** You MUST NOT begin or continue writing the spec while unresolved questions remain. You must exhaust all questions and collect convergence parameters before starting. If you discover new ambiguities while writing the spec, pause and return to the interview phase.

## Scope

### In Scope

1. SKILL.md interview phase (Workflow A, Step 0) — §1.1-§1.5
2. Convergence parameter collection during interview exit — §1.2
3. `workflow.review_clean_rounds` and `workflow.audit_clean_rounds` fields in TaskFile — §2.1
4. Validation of new workflow fields in `tp validate` (range 1-10, deserialization defaults) — §2.2
5. `tp init` writes both fields with default value 2 — §2.2
6. `tp set --workflow` subcommand form for file-level workflow fields — §2.4
7. SKILL.md restructuring into SKILL.md (90-120 lines) + REFERENCE.md — §3
8. Updated skill description with trigger phrases — §3.3
9. Convergence enforcement language (verbatim blocks) in SKILL.md — §4

### Out of Scope

- HITL tagging or human-in-the-loop mechanisms (tp's philosophy: front-load all decisions)
- CLI enforcement of review/audit round counting — convergence parameters are advisory data; enforcement is the agent's responsibility via SKILL.md language. This is a known tradeoff: agents using tp CLI directly without the skill are not bound by convergence rules.
- Interview transcript storage or persistence
- Changes to `tp review` or `tp audit` CLI commands
- Vertical slice / tracer bullet terminology changes
- Code changes to `cli/plan.go` — workflow passthrough already works (§2.3)

## Migration

- Existing task files without `review_clean_rounds` / `audit_clean_rounds`: defaults (2) applied at deserialization time via custom `UnmarshalJSON` when the key is absent. Explicit `0` in JSON is NOT coerced — it fails range validation.
- Existing consumers of `tp plan` JSON output will now see `review_clean_rounds` and `audit_clean_rounds` in the `workflow` block.
- No breaking changes — new fields are optional with sensible defaults.
- `tp import` and `tp add` accept existing task files unchanged.
- `tp validate --strict` does not flag missing convergence fields.
- `tp validate` JSON output gains a `workflow` key. Consumers that strict-parse the output should expect this new key. Existing keys (`valid`, `errors`, `warnings`) are unchanged.

## Implementation Notes

### CLI Changes

1. **model/taskfile.go**: Add `ReviewCleanRounds int` and `AuditCleanRounds int` to `Workflow` struct with `json:"review_clean_rounds"` and `json:"audit_clean_rounds"` tags (no `omitempty`). Add custom `UnmarshalJSON` on `Workflow` using `*int` pointers internally to distinguish absent (nil → default 2) from explicit-zero (non-nil, value 0 → preserved for validation). After deserialization, the struct fields are plain `int` with resolved values.
2. **engine/validate.go**: Validate both fields are in range 1-10 using parameterized error `"<field> must be between 1 and 10"`. Explicit 0 is a validate-time error only (does not break other commands). Add `workflow` key to `ValidationResult` output (always included, shows post-deserialization values).
3. **cli/init.go**: Set `ReviewCleanRounds: 2` and `AuditCleanRounds: 2` in the generated `Workflow` struct.
4. **cli/set.go**: Add `--workflow` flag. When set, parse `field=value` pairs and update `TaskFile.Workflow` instead of a task. Editable fields: `review_clean_rounds`, `audit_clean_rounds` only. Validate range 1-10. Reject unknown fields. Require at least one pair. Output: `{"updated": {...}}` on success, `{"error": "..."}` on failure.
5. _(no change needed in cli/plan.go — workflow passthrough already works)_

### Skill File Changes (Primary)

1. **skills/tp/SKILL.md**: Rewrite to 90-120 lines. All 8 interview steps represented, convergence enforcement verbatim from §4, condensed workflows with forward-references to enforcement section, decomposition rules, closure rules, REFERENCE.md pointer.
2. **skills/tp/REFERENCE.md**: New file with 9 sections (ordered per §3.2 "Moved to REFERENCE.md" list). Content migrated verbatim from current SKILL.md sections indicated. Editorial reformatting (fixing typos, updating section numbers) is permitted but content additions or deletions require a separate spec.
