# Review Perspectives: Documentation & Test Planning

**Status:** Draft
**Date:** 2026-04-02

## 1. Problem

`tp review` generates 3 adversarial prompts (implementer, tester, architect) that review spec **quality**. But after a spec passes review and enters implementation, two critical planning steps are missing:

1. **Documentation planning** — What documentation files need to be created or updated? Where does the new content go? What existing content drifts?
2. **Test planning** — What test cases are needed? What edge cases? What fixtures? Where do they live?

Today, the agent figures these out ad-hoc during implementation. This means:
- Documentation updates are discovered late, causing rework
- Test gaps are found after implementation, not before
- No structured plan exists for "what docs/tests to add" — the agent improvises
- Same spec reviewed by different agents produces inconsistent doc/test plans

## 2. Proposed Solution

Add `--perspective` flag to `tp review`:

```bash
tp review spec.md                                    # Existing: spec quality (implementer/tester/architect)
tp review spec.md --perspective documentation         # New: doc change plan
tp review spec.md --perspective documentation --docs-path docs/
tp review spec.md --perspective testing               # New: test plan
tp review spec.md --perspective testing --test-path internal/
```

Each perspective generates a single, targeted prompt. The agent feeds this prompt to a sub-agent which produces a structured JSON plan.

tp generates the prompt (with spec content + doc/test structure context). The agent executes it (via AI). P4: Agent Plans, Tool Executes.

## 3. Design Principles

| # | Principle | Application |
|---|-----------|-------------|
| P1 | AX First | One flag, one prompt per perspective. Agent gets actionable plan in one call |
| P2 | Batch Parity | Works for 1-file docs or 100-file docs. Same output shape |
| P3 | Minimal Tokens | Only files matching spec terms included in context. Structure map is compact tree |
| P4 | Agent Plans, Tool Executes | tp generates structured prompt with doc structure. Agent executes and produces plan |

## 4. Command Changes

### 4.1 New Flags

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--perspective` | string | "" | Review perspective: `documentation` or `testing`. Empty = existing behavior |
| `--docs-path` | string | "" | Path to documentation directory (required when perspective=documentation) |
| `--test-path` | string | "" | Path to test directory (required when perspective=testing) |

### 4.2 Validation Rules

1. `--perspective` accepts only `documentation` or `testing`. Error (exit 2) otherwise.
2. `--docs-path` is required when `--perspective=documentation`. Error if missing.
3. `--test-path` is required when `--perspective=testing`. Error if missing.
4. `--docs-path` must be an existing directory. Error (exit 3) if not.
5. `--test-path` must be an existing directory. Error (exit 3) if not.
6. `--perspective` is mutually exclusive with `--round`/`--findings`. Error (exit 2) if both provided.
7. When `--perspective` is empty: existing behavior unchanged (3 prompts, no path required).

### 4.3 Behavior Matrix

| `--perspective` | `--round` | `--findings` | Behavior |
|----------------|-----------|-------------|----------|
| (empty) | any | any | Existing: 3 adversarial prompts |
| `documentation` | — | — | Doc change plan prompt |
| `testing` | — | — | Test plan prompt |
| `documentation` | 1 | file | Error: mutually exclusive |
| `invalid` | — | — | Error: invalid perspective |

## 5. Documentation Perspective

### 5.1 Context Collection

When `--perspective=documentation` and `--docs-path=docs/`:

1. Walk `docs-path` recursively, collect all `.md` files
2. Build a structure map: directory tree with file names (no content yet)
3. Grep each file for terms extracted from spec headings (## and ### text)
4. Rank files by match count, select top 15 files + all index.md files + all config files
5. Read content of selected files (cap each at 5000 chars, total at 30000 chars)
6. Include structure map + selected file contents in prompt context

### 5.2 Structure Map Format

```
Documentation structure:
docs/
├ index.md
├ guide/
│ ├ getting-started.md
│ ├ workflow.md
│ └ commit.md
└ reference/
  ├ commands.md
  └ cli.md
```

Plain tree output from `find` or equivalent. No VitePress-specific parsing.

### 5.3 Documentation Prompt

Role: "You are a technical writer planning documentation changes for a new feature."

The prompt contains:
1. Role instruction
2. Spec content (truncated at 10000 chars, same as existing)
3. Structure map of documentation directory
4. Content of documentation files matching spec terms (capped)
5. Systematic checklist (6 analysis steps, see 5.4)
6. Output format specification (JSON schema)

### 5.4 Documentation Analysis Checklist

The checklist embedded in the prompt:

```
Analyze the spec against the existing documentation. For each step below,
if an issue is found, produce a plan item.

A1. COMPLETENESS — Spec-to-Doc Coverage
For each ## and ### heading in the spec:
- Does a corresponding doc page or section exist?
- If no → plan "create-page" or "update-section"

A2. TERM COVERAGE — Commands, Flags, Config Keys
For each command name, CLI flag, config key, type name, or new concept in the spec:
- Is it documented anywhere in the existing docs?
- If no → plan "update-section" in the page covering the same domain

A3. DRIFT — Accuracy of Existing Content
For each existing doc page that covers the same domain as the spec:
- Does any statement contradict the spec?
- Are code examples still valid per the spec?
- If yes → plan "fix-drift"

A4. NAVIGATION — Structure and Discovery
- Does each new or updated page have a place in the navigation structure?
- Should any index page be updated to reference the new content?
- If yes → plan "update-config" or "update-index"

A5. CROSS-REFERENCES — Link Integrity
- Do related existing pages reference the new feature area?
- Does the new content reference back to related pages?
- If no → plan "add-crossref"
```

### 5.5 Documentation Plan Output Format

The prompt requests the sub-agent output as NDJSON (one JSON object per line):

```json
{"id":"doc-001","action":"create-page","file":"docs/guide/batch-closing.md","location":null,"spec_ref":"### 2.3 Batch Closing","description":"New page for batch closing workflow","detail":{"outline":["## Overview","## Quick Start","## NDJSON Format","## Examples"]},"priority":"must","depends_on":[]}
{"id":"doc-002","action":"update-section","file":"docs/guide/getting-started.md","location":"## Commands","spec_ref":"### 2.3 Batch Closing","description":"Add tp done --batch to command list","detail":{"content":"Add row: | `tp done --batch` | Close multiple tasks from NDJSON file |"},"priority":"must","depends_on":["doc-001"]}
{"id":"doc-003","action":"fix-drift","file":"docs/reference/commands.md","location":"## tp done","spec_ref":"### 2.3 Batch Closing","description":"Current docs say single-task only, batch is now supported","detail":{"current":"tp done <id> <reason> — closes a single task","target":"tp done <id> <reason> — closes a single task. tp done --batch <file> — closes multiple tasks from NDJSON."},"priority":"must","depends_on":[]}
{"id":"doc-004","action":"update-config","file":"docs/.vitepress/config.ts","location":"sidebar","spec_ref":"### 2.3 Batch Closing","description":"Add Batch Closing entry to sidebar","detail":{"entry":"{ text: 'Batch Closing', link: '/guide/batch-closing' }","after":"Commit"},"priority":"must","depends_on":["doc-001"]}
{"id":"doc-005","action":"add-crossref","file":"docs/guide/workflow.md","location":"## Closing Tasks","spec_ref":"### 2.3 Batch Closing","description":"Add See also link to batch closing page","detail":{"link":"[Batch Closing](/guide/batch-closing)"},"priority":"should","depends_on":["doc-001"]}
{"id":"doc-006","action":"update-index","file":"docs/index.md","location":"## Features","spec_ref":"### 2.3 Batch Closing","description":"Add batch closing to feature list","detail":{"content":"Add: - Batch close multiple tasks in one call"},"priority":"should","depends_on":["doc-001"]}
```

### 5.6 Documentation Action Types

| Action | Description | When |
|--------|-------------|------|
| `create-page` | New documentation file needed | Feature is a new concept/workflow/area |
| `update-section` | Add content to existing page | New flag, config option, or detail within existing topic |
| `fix-drift` | Correct existing content contradicting spec | Spec changes existing behavior or adds capabilities |
| `update-config` | Update navigation/sidebar config | New page needs to be discoverable |
| `add-crossref` | Add links between pages | Related pages should reference each other |
| `update-index` | Update index/landing page | Index pages list features or summarize sections |

### 5.7 Documentation Priority Levels

| Priority | Meaning |
|----------|---------|
| `must` | Without this, documentation is incorrect or incomplete |
| `should` | Significantly improves documentation quality |
| `could` | Nice to have, low impact |

### 5.8 Documentation Plan Review Object

The `tp review --perspective=documentation` output wraps the prompt in the same structure as existing review:

```json
{
  "spec": "spec/0.13.0-review-perspectives.md",
  "perspective": "documentation",
  "docs_path": "docs/",
  "docs_structure": {
    "total_files": 7,
    "reviewed_files": 5,
    "structure_map": "docs/\n├ index.md\n├ guide/\n│ ├ getting-started.md\n..."
  },
  "prompts": [
    {
      "role": "documentation-planner",
      "category": "completeness",
      "prompt": "You are a technical writer planning documentation changes..."
    }
  ],
  "review_loop": {
    "round": 1,
    "max_rounds": 1,
    "convergence": "single-pass plan generation",
    "previous_findings": 0,
    "instruction": "Spawn a sub-agent with this prompt. Collect the NDJSON plan. Review the plan for completeness, then append the plan to the spec as a '## Documentation Plan' section."
  }

```

Note: Documentation and test planning are single-pass (no multi-round). The output is a plan, not findings. Multi-round doesn't make sense here — the plan is either complete or the spec is wrong.

## 6. Testing Perspective

### 6.1 Context Collection

When `--perspective=testing` and `--test-path=internal/`:

1. Walk `test-path` recursively, collect all `_test.go` files (or `.test.ts`, `.spec.ts`, etc.)
2. Build a structure map: test files grouped by the source file they test
3. Read content of existing test files (cap total at 20000 chars)
4. Include structure map + test file contents in prompt context

### 6.2 Testing Prompt

Role: "You are a QA engineer planning test coverage for a new feature."

The prompt contains:
1. Role instruction
2. Spec content (truncated at 10000 chars)
3. Structure map of test directory
4. Content of existing test files covering the same source modules (capped)
5. Systematic checklist (6 analysis steps, see 6.3)
6. Output format specification (JSON schema)

### 6.3 Testing Analysis Checklist

```
Analyze the spec and plan the test coverage needed. For each step below,
if a test is needed, produce a plan item.

T1. ACCEPTANCE CRITERIA COVERAGE
For each numbered list item (acceptance criteria) in the spec:
- Can this be verified with a test?
- What test function would verify it?
- If testable → plan "create-test"

T2. TABLE ROW COVERAGE
For each row in each table in the spec:
- Is there a test that exercises this row's behavior?
- If no → plan "create-test"

T3. HAPPY PATH
- What is the primary use case / happy path?
- Is it covered by existing tests or does it need a new test?
- → plan "create-test"

T4. ERROR PATHS
- What can go wrong? What error conditions exist?
- File not found, invalid input, permission denied, empty input, etc.
- → plan "create-test" for each distinct error path

T5. EDGE CASES
- Boundary conditions (empty, single item, very large input)
- Concurrent access, race conditions (if applicable)
- Zero-value / nil / default behavior
- → plan "create-test" for each edge case

T6. INTEGRATION POINTS
- Does the feature interact with other features or external systems?
- Are there integration tests needed?
- → plan "create-test" or "update-test"

T7. FIXTURE / HELPER NEEDS
- Are new test fixtures, mock data, or helper functions needed?
- → plan "create-fixture"
```

### 6.4 Test Plan Output Format

NDJSON, one object per line:

```json
{"id":"test-001","action":"create-test","file":"internal/cli/review_test.go","location":"TestReviewDocumentationPerspective","spec_ref":"### 4.1 New Flags","description":"Test --perspective documentation with --docs-path","detail":{"type":"unit","assertions":["exit code 0 on valid docs-path","prompt contains documentation structure","prompt contains spec content"],"inputs":{"docs_path":"tmp_dir_with_md_files"}},"priority":"must","depends_on":[]}
{"id":"test-002","action":"create-test","file":"internal/cli/review_test.go","location":"TestReviewTestingPerspective","spec_ref":"### 4.1 New Flags","description":"Test --perspective testing with --test-path","detail":{"type":"unit","assertions":["exit code 0 on valid test-path","prompt contains test structure map","prompt contains spec content"],"inputs":{"test_path":"tmp_dir_with_test_files"}},"priority":"must","depends_on":[]}
{"id":"test-003","action":"create-fixture","file":"internal/cli/testdata/review_docs/","location":null,"spec_ref":"### 5.1 Context Collection","description":"Fixture: sample docs directory for documentation perspective tests","detail":{"files":["index.md","guide/getting-started.md","reference/api.md"]},"priority":"should","depends_on":[]}
{"id":"test-004","action":"create-test","file":"internal/cli/review_test.go","location":"TestReviewPerspectiveInvalid","spec_ref":"### 4.2 Validation Rules","description":"Test invalid perspective value returns error","detail":{"type":"unit","assertions":["exit code 2","stderr contains 'invalid perspective'"],"inputs":{"perspective":"invalid"}},"priority":"must","depends_on":[]}
```

### 6.5 Test Action Types

| Action | Description | When |
|--------|-------------|------|
| `create-test` | New test function needed | New behavior to verify |
| `update-test` | Existing test needs modification | Spec changes existing behavior |
| `create-fixture` | New test data/helper needed | Tests need mock data or file fixtures |

## 7. Implementation

### 7.1 Output Structure

```go
type docStructure struct {
    TotalFiles    int    `json:"total_files"`
    ReviewedFiles int    `json:"reviewed_files"`
    StructureMap  string `json:"structure_map"`
}

type perspectiveResult struct {
    Spec          string         `json:"spec"`
    Perspective   string         `json:"perspective"`
    DocsPath      string         `json:"docs_path,omitempty"`
    TestPath      string         `json:"test_path,omitempty"`
    DocsStructure *docStructure  `json:"docs_structure,omitempty"`
    TestStructure *docStructure  `json:"test_structure,omitempty"`
    Prompts       []reviewPrompt `json:"prompts"`
    ReviewLoop    reviewLoop     `json:"review_loop"`
}
```

When `--perspective` is empty (existing behavior), the existing `reviewResult` struct is used unchanged. The two structs share the same JSON keys (`spec`, `prompts`, `review_loop`) so parsers expecting the old format continue to work. The only difference is `prompts` has 1 item instead of 3 when a perspective is active.

### 7.2 Functions

1. `walkDocTree(path string) (tree string, files []string)` — Build structure map + file list
2. `rankFilesBySpecTerms(files []string, specLines []string, max int) []string` — Grep-based relevance ranking
3. `readFilesContent(files []string, maxPerFile, maxTotal int) map[string]string` — Read capped file contents
4. `generateDocPlanPrompt(specContent, structureMap, fileContents map[string]string) reviewPrompt`
5. `generateTestPlanPrompt(specContent, structureMap, fileContents map[string]string) reviewPrompt`
6. `parsePerspective(perspective string) error` — Validate perspective value

### 7.3 Changes to Existing Code

1. `newReviewCmd()`: Add `--perspective`, `--docs-path`, `--test-path` flags
2. `runReview()`: Branch on perspective:
   - Empty → existing behavior (3 prompts)
   - `documentation` → single prompt, doc context
   - `testing` → single prompt, test context
3. Mutual exclusivity check: `--perspective` + `--round`/`--findings` = error

## 8. Implementation Order

1. Add `--perspective`, `--docs-path`, `--test-path` flags to `newReviewCmd()`
2. Add perspective validation (valid values, required paths, mutual exclusivity)
3. Implement `walkDocTree()` — directory tree structure map
4. Implement `rankFilesBySpecTerms()` — grep-based file selection
5. Implement `readFilesContent()` — capped file reading
6. Implement `generateDocPlanPrompt()` — documentation planning prompt
7. Implement `generateTestPlanPrompt()` — test planning prompt
8. Wire perspective branch into `runReview()`
9. Add `perspectiveResult` output structure
10. Write tests for documentation perspective
11. Write tests for testing perspective
12. Write tests for validation (invalid perspective, missing path, mutual exclusivity)

## 9. Test Cases

1. `--perspective documentation` with valid `--docs-path`: Returns single prompt with doc structure map
2. `--perspective testing` with valid `--test-path`: Returns single prompt with test structure map
3. `--perspective invalid`: Exit code 2, error message
4. `--perspective documentation` without `--docs-path`: Exit code 2, error message
5. `--perspective documentation` with non-existent `--docs-path`: Exit code 3, error message
6. `--perspective documentation --round 2`: Exit code 2, mutual exclusivity error
7. `--perspective documentation --findings file.ndjson`: Exit code 2, mutual exclusivity error
8. Empty `--perspective` (no flag): Existing 3-prompt behavior unchanged
9. Documentation prompt contains structure map: Structure map present in prompt text
10. Documentation prompt contains spec content: Spec text present in prompt text
11. Documentation prompt contains file contents: Files matching spec terms included
12. Documentation prompt contains analysis checklist: All 5 checklist items (A1-A5) present
13. Test prompt contains structure map: Test directory tree present in prompt text
14. Test prompt contains analysis checklist: All 7 checklist items (T1-T7) present
15. Empty docs directory: Prompt still works with empty structure map, no panic
16. Docs directory with many files (20+): Capped at 15 reviewed files

## 10. What Doesn't Change

- Existing `tp review` without `--perspective` is unchanged
- `tp lint` unchanged
- `tp validate` unchanged
- No new dependencies (uses `os`, `path/filepath`, `bufio`)
- No AI/LLM calls from tp — still generates prompts only
- Task lifecycle unchanged
- `--round` and `--findings` unchanged for spec quality review
